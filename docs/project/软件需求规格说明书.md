# 1 功能需求
本模块围绕软件**“基础功能复刻+环保社交增值”**双层架构设计，覆盖拼车乘客、私家车主、管理员三类核心用户的全流程操作诉求，严格复刻高德顺风车核心底层能力，同步落地安全防护、环保创新、社交差异化、合规运营及无障碍/老年适配等关键功能；所有功能按落地优先级划分为**必做**（核心刚需、决议强制项）、**选做**（增值优化、后续迭代）两类，保障拼车场景闭环、用户体验流畅、运营合规可控。

## 1.1 核心用户角色功能
针对系统三类核心用户，分别配置专属操作权限与功能模块，实现角色权责清晰、操作流程专属化，核心复刻高德顺风车双向匹配、行程管控等基础能力，避免功能重复开发：

### 1.1.1 拼车乘客功能
+ **账号与基础信息管理【必做】**：支持注册、登录、找回密码，完善个人基础属性（年龄、性别、国籍、无障碍需求等），可自主编辑个人信息、设置紧急联系人等。
+ **拼车需求发布与双向筛选【必做】**：填写出发地、目的地、出行时间、乘车人数等信息发布拼车单；依托智能路径匹配算法，自动筛选路线重合度高的车主；可自主查看并筛选车主出发时间、路线重合度、信用评级、历史接单数、车辆型号、车主评价标签、服务口碑等信息，自主选择意向接单车主，完成司乘双向筛选。
+ **行程管控与安全感知【必做（资质核验）+选做（轨迹监控与行程共享）】**：核验车主资质与实名认证信息为必做；实时导航轨迹查看、路线偏移自动警报、紧急联系人行程共享为选做功能。
+ **环保积分与福利兑换【选做】**：参与拼车积累碳减分值，自动计入碳积分系统；通过积分兑换平台优惠券；查看登录界面成长树可视化状态，用车次数越多树木生长越茂盛。
+ **社交与评价功能【必做（基础评价）+选做（社交模式）】**：基础评价为必做，行程结束后对车主进行评分；选做拓展社交型（e人车）/效率型（i人车）匹配模式，添加健谈、安静等个性化标签，查看历史行程、评价记录。
+ **费用与合规功能【必做】**：使用高速费分摊计算器自动核算分摊金额；接收地域化运营规则实时提示，查看订单合规相关说明。

### 1.1.2 私家车主功能
+ **资质认证与账号管理【必做】**：完成平台实名认证，提交驾驶证、行驶证等资质证明审核；完善个人信息、车辆信息、信用档案；开启联系方式加密传输，保障隐私安全。
+ **订单接单与双向匹配【必做】**：查看周边拼车订单、乘客出行信息（出发地、目的地、出行时间、乘车人数）；通过路径匹配算法接收精准订单推送；可自主筛选乘客出行需求匹配度、乘车人数、出行时段等信息，自主选择是否接单，确认接单后生成正式行程订单。
+ **行程执行与安全监控【必做】**：遵守每日接单上限（2-4单）地域化规则；同步行程信息至乘客与平台；导航轨迹实时上传、路线偏移自动触发警报为【选做】。
+ **信用与评价管理【必做】**：查看个人信用评级、历史接单数等信息；接收乘客评价与标签；维护个人服务口碑。
+ **费用核算功能【必做】**：通过高速费分摊计算器生成分摊明细，与乘客确认费用；查看行程收益、历史账单。

### 1.1.3 管理员功能
+ **用户与资质管理【必做】**：审核车主驾驶证、行驶证等资质信息；管理三类用户账号，处理账号异常、投诉申诉；维护用户属性数据库，统计用户特征数据。
+ **订单与行程管控【必做】**：监控实时行程轨迹，处理路线偏移警报、异常订单；管控地域化订单匹配，屏蔽不符合当地准入政策的订单。
+ **合规与规则配置【必做】**：更新不同城市准入政策、每日接单上限等地域化运营规则；配置平台功能规则、隐私协议，通过问号图标实现规则实时提示。
+ **数据与积分管理【必做（基础数据）+选做（积分运营）】**：基础数据监控、统计为必做；碳积分系统、成长树数据运营、优惠券管理为选做。
+ **安全机制管控【必做】**：监控导航轨迹监控系统运行状态；管理用户信息加密传输、隐私保护机制；处理安全相关异常事件。

## 1.2 安全防护核心功能
+ **导航轨迹监控警报【选做】**：行车过程中实时上传导航轨迹，检测到路线异常偏移时，自动向乘客、车主、平台后台发送警报通知。
+ **用户隐私加密保护【必做】**：搭建专属信息加密传输方案，加密保护车主联系方式等敏感信息，杜绝信息泄露。
+ **实名认证与资质核验【必做】**：参照网约车平台体系，强制车主完成驾驶证、行驶证资质审核，乘客完成基础实名认证，从源头把控用户资质。

## 1.3 特色创新功能
### 1.3.1 环保理念相关功能
+ **碳积分累计系统【选做】**：根据拼车减碳量自动核算积分，乘客累计积分可兑换平台优惠券、权益福利。
+ **成长树可视化激励【选做】**：登录界面展示专属成长树，用户每完成一次拼车，成长树获得生长能量，用车频次越高树木形态越茂盛，打造游戏化激励体验。

### 1.3.2 社交差异化功能
+ **双模式匹配机制【选做】**：提供社交型（e人车）、效率型（i人车）两种匹配模式，社交模式鼓励司乘互动，效率模式仅保留基础行程沟通，降低互动干扰。
+ **个性化评价标签【选做】**：支持司乘双方添加健谈、安静、守时等个性化标签，形成标签体系，为后续匹配提供参考。

## 1.4 合规运营功能
+ **地域化规则适配【必做】**：内置不同城市准入政策、运营规则，自动屏蔽不符合当地法规的订单匹配；通过问号图标实时展示所有运营规则，无隐藏条款。
+ **接单上限管控【必做】**：按地域规则设置车主每日接单上限（2-4单），系统自动限制超额接单。
+ **高速费分摊计算【必做】**：内置高速费分摊计算器，自动核算行程高速费，生成清晰分摊明细供司乘确认。
+ **紧急联系人通知【选做】**：行程启动后，自动向乘客预设紧急联系人同步行程信息，保障出行安全。

## 1.5 无障碍与差异化功能
+ **无障碍模式【选做】**：作为核心差异化功能，适配残障用户操作习惯，支持语音播报、大字模式、简易操作流程等无障碍适配方案。
+ **老年人模式【选做】**：针对老年用户专项优化，支持一键开启全局超大字体、简化页面布局、删减冗余交互入口、仅保留拼车核心功能，降低老年群体操作门槛，适配视力衰退、操作不熟练的使用场景。
+ **多语言支持【选做】**：适配不同国籍用户，提供多语言切换功能，覆盖外籍用户使用需求。

## 1.6 **异常业务流处理功能**
+ **订单超时未接单【必做】**：乘客发布行程后，若在期望出发时间前15分钟仍无车主接单，系统需自动取消该订单，并向乘客推送提醒消息。
+ **中途取消与违约处理【必做】**：
    - 车主接单后、行程开始前，若任意一方无故取消订单，系统将记录违约行为，扣除其信用评级分数（或扣除部分违约金）。
    - 行程中途若因车辆故障等不可抗力导致无法继续，车主需在系统点击“行程异常终止”，系统按已行驶里程按比例重新核算费用，剩余费用原路退还乘客。
+ **支付异常处理【必做】**：若到达目的地后乘客因网络或余额问题支付失败，系统将限制其后续发布新行程的权限，直至补缴欠款。

# 2 性能需求
本章节针对移动端私家车拼车软件的轻量化运营场景，结合产品定位与实际使用诉求，对各项性能指标进行**合理、务实的规划测算**，采用适配中小规模用户的性能标准，不追求商用级超高并发配置，核心覆盖规模大小、响应效率、并发能力、计算精度、可靠性与安全性六大维度，保障软件稳定运行、用户体验流畅。

## 2.1 规模大小约束
### 2.1.1 存储容量限制
+ **单用户数据存储**：乘客/车主基础信息、资质文件、行程记录等单用户数据总量≤50MB，其中文字类信息≤50KB，证件图片类压缩后≤50MB。
+ **订单数据存储**：单条拼车订单数据（含行程轨迹、费用、评价）≤20KB，项目初期总订单存储量上限≤10000条，整体数据库存储容量≤500MB。
+ **日志与缓存存储**：系统运行日志、接口缓存数据总量≤200MB，支持定期自动清理冗余数据，避免存储溢出。

### 2.1.2 整体数据规模
项目初期轻量化运营阶段，系统注册用户总量上限≤5000人，日均活跃用户≤500人，单日生成订单总量≤200单，聚焦中小规模用户群体的拼车服务支撑。

## 2.2 执行速度与响应时间
+ **页面加载速度**：APP所有功能页面冷启动加载≤3秒，热启动切换页面≤1.5秒，老年人/无障碍模式页面加载≤2秒。
+ **接口响应时间**：登录、注册、订单发布、筛选查询等核心接口响应≤1秒；路径匹配、订单推送等算法类接口响应≤2秒；无长时间卡顿、无响应场景。
+ **操作反馈速度**：按钮点击、开关切换（如老年人模式、无障碍模式）、评价提交等即时操作，反馈时间≤0.5秒。

## 2.3 吞吐量与并发个数
+ **并发用户数**：系统支持**峰值同时在线用户≤500人**，核心功能模块（订单、筛选、支付）并发操作用户≤100人，满足轻量化运营阶段的并发使用需求。
+ **系统吞吐量**：每秒处理接口请求数≤20次，每秒完成订单匹配/接单操作≤5次，无请求阻塞、数据丢失情况。
+ **并发容错**：达到并发上限时，系统提示“当前操作人数较多，请稍后重试”，不出现崩溃、闪退问题。

## 2.4 计算精度
+ **费用计算精度**：拼车费用、高速分摊费、优惠券抵扣等金额计算，保留**小数点后2位**，无计算误差、金额错乱问题。
+ **里程与时间精度**：导航里程计算保留小数点后1位（单位：公里），行程时间计算精确到分钟，碳积分核算取整数，无精度偏差。
+ **匹配算法精度**：路线重合度计算误差≤5%，满足拼车匹配的基础准确性即可。

## 2.5 可靠性要求
+ **系统可用率**：系统正常运行可用率≥95%，避免频繁崩溃、闪退，保障日常使用稳定性。
+ **故障恢复能力**：遇到网络中断、程序异常等故障，重启后10秒内恢复运行，已保存数据不丢失。
+ **数据可靠性**：用户信息、订单记录、行程数据存储成功率≥99.9%，无数据丢失、篡改情况。

## 2.6 安全性性能
+ **加密处理速度**：用户手机号、证件信息等敏感数据加密/解密处理≤0.5秒，不影响正常操作流程。
+ **安全拦截效率**：异常订单、违规资质、非法请求的识别拦截≤1秒，保障平台基础安全。
+ **隐私防护**：用户敏感信息全程加密传输，无信息泄露、越权查看的性能漏洞。

# 3 环境需求
本章明确软件运行所需的硬件支撑条件与软件适配环境，结合**前端React Native（RN）、后端Node.js**技术栈选型，覆盖乘客/车主移动端、后台管理端、服务端全场景，保障软件正常编译、部署、运行与迭代维护。

## 3.1 硬件环境
硬件环境分为移动端用户硬件、服务端部署硬件、管理端操作硬件三类，分别适配不同角色使用场景，满足基础运行与性能承载要求。

### 3.1.1 移动端用户硬件（乘客/车主）
+ **移动设备类型**：支持智能手机（Android手机、iPhone），兼容平板设备（可选），满足手持便携、随时随地发起/承接拼车订单的使用场景。
+ **最低硬件配置**：运行内存≥2GB，机身存储剩余空间≥1GB，处理器为四核及以上，具备正常的GPS定位、网络通信、触控交互功能，保证APP流畅安装与运行。
+ **外设依赖**：需支持移动网络（4G/5G）或Wi-Fi连接，依赖GPS模块实现定位、轨迹监控与路径匹配功能。

### 3.1.2 服务端硬件
+ **服务器类型**：支持云服务器（阿里云、腾讯云等）或物理服务器，满足后端Node.js服务部署、数据库存储、接口转发需求。
+ **基础配置要求**：CPU≥2核，运行内存≥4GB，存储容量≥40GB（系统盘+数据盘），具备公网IP与稳定带宽，支持7×24小时不间断运行。
+ **网络要求**：上行/下行带宽≥2Mbps，保障接口响应、数据传输、轨迹上传的网络稳定性。

### 3.1.3 管理端硬件（管理员）
+ **设备类型**：台式电脑、笔记本电脑，用于后台管理系统操作、数据监控与资质审核。
+ **最低配置**：运行内存≥4GB，存储空间≥10GB，具备稳定网络连接，支持浏览器正常访问后台管理页面。

## 3.2 软件环境
软件环境贴合项目技术栈选型，分为客户端运行环境、服务端运行环境、开发编译环境三类，明确版本适配与依赖组件。

### 3.2.1 客户端运行环境
+ **Android端**：支持Android 8.0及以上版本，兼容主流Android定制系统，依托React Native框架实现跨端渲染，无需额外安装插件即可运行APP。
+ **iOS端**：支持iOS 13.0及以上版本，适配iPhone全系机型，符合Apple应用运行规范，依托RN框架实现功能交互。
+ **权限依赖**：需获取设备定位权限、网络权限、存储权限、电话权限（可选），保障拼车核心功能正常调用。

### 3.2.2 服务端运行环境
+ **后端核心环境**：Node.js 14.0及以上稳定版本，搭配npm/yarn包管理工具，支撑后端接口开发、业务逻辑处理与并发请求响应。
+ **数据库环境**：支持MySQL 5.7及以上版本（关系型数据存储）、Redis 5.0及以上版本（缓存与会话管理），实现用户数据、订单数据、轨迹数据的持久化存储。
+ **中间件与服务**：Nginx（反向代理与负载均衡）、PM2（Node.js进程守护），保障后端服务稳定运行、故障自动重启。
+ **操作系统**：服务器端支持CentOS 7.x、Ubuntu 20.04及以上Linux发行版，兼容Windows Server 2019（可选）。

### 3.2.3 开发编译环境
+ **前端开发环境**：React Native 0.68及以上版本，Node.js开发环境，搭配Android Studio/Xcode用于移动端编译调试，VS Code等代码编辑器。
+ **后端开发环境**：Node.js开发套件，Postman/Swagger接口测试工具，数据库可视化管理工具（Navicat、Redis Desktop Manager）。
+ **通用依赖**：Git代码版本控制工具，保障项目代码协同管理与迭代更新。

# 4 界面需求
本章围绕软件界面交互、外部系统数据交互、数据格式及存储介质展开要求，基于React Native（RN）跨端框架实现界面统一渲染，兼顾普通用户、老年人、无障碍用户的操作习惯，同时明确内外系统数据流转规则，保证界面易用性、规范性与数据流转顺畅性。

## 4.1 外部系统输入（界面数据来源）
界面渲染、功能交互所需的核心数据，存在多处外部系统输入，均为界面正常运行的必要支撑，具体如下：

+ **地图导航系统输入【必做】**：接入第三方地图SDK（高德地图），输入实时定位数据、路径规划数据、路线偏移判定数据、地理编码数据，支撑拼车路径匹配、行程轨迹展示、导航监控界面渲染。
+ **实名认证资质系统输入【选做】**：对接第三方实名认证接口，输入车主驾驶证、行驶证核验结果，乘客身份核验结果，资质审核状态同步至用户中心、订单管理界面；平台内资质审核为【必做】。
+ **多语言词典系统输入【选做】**：调用多语言配置接口，输入不同语种的文本词条、翻译内容，适配界面多语言切换功能。
+ **权益积分系统输入【选做】**：碳积分、优惠券数据由后台权益系统输入，同步至个人中心、环保成长树界面，展示积分余额与兑换状态。

## 4.2 外部系统输出（界面数据外传）
软件界面产生的业务数据、操作数据，需定向输出至外部系统，实现数据互通与闭环管理，具体如下：

+ **行程监控数据输出【必做】**：将实时行程轨迹、路线偏移警报信息，输出至平台后台监控系统、乘客紧急联系人终端，同步展示于行程安全界面。
+ **司乘交互数据输出【必做】**：将双向筛选指令、订单接单/取消操作、评价打分数据，输出至订单匹配系统、信用评级系统，更新司机名录、订单列表界面状态。
+ **合规运营数据输出【必做】**：将车主接单次数、地域订单匹配记录，输出至地域合规管控系统，界面自动屏蔽违规订单、提示运营规则。
+ **环保行为数据输出【选做】**：将拼车减碳数据、成长树生长进度，输出至碳积分核算系统，同步更新界面可视化展示效果。

## 4.3 数据格式规定
为保证界面渲染稳定、数据交互无异常，对内外流转数据、界面展示数据制定统一格式规范：

+ **界面渲染数据格式【必做】**：采用JSON格式传输界面文本、图片、状态数据；定位、里程、金额等数值型数据采用数字格式，文本类数据采用UTF-8编码，避免乱码；图片资源采用WebP/JPG格式，压缩后适配界面加载速度要求。
+ **交互指令数据格式【必做】**：司乘筛选、订单操作、模式切换（老年人/无障碍）等交互指令，采用固定字段的JSON格式，字段名称、参数类型统一规范，杜绝格式错乱导致界面无响应。
+ **外部接口数据格式【必做】**：与地图、实名认证、后台系统对接，严格遵循接口约定的请求/响应格式，必填字段、数据类型、返回码统一标准，界面可正常解析并展示异常提示。
+ **展示类数据格式【必做】**：金额保留小数点后2位，里程保留1位小数，时间统一采用YYYY-MM-DD HH:MM格式，信用评级、接单量采用纯数字/字母标识，界面展示规整统一。

## 4.4 数据存储介质规定
针对界面缓存数据、临时交互数据、持久化业务数据，明确存储介质要求，兼顾界面响应速度与数据安全性：

+ **客户端临时缓存介质【必做】**：界面常用配置（语言、模式、布局）、近期行程记录等临时数据，存储于移动端设备内置闪存、RAM缓存，采用RN框架自带缓存机制，定期自动清理过期缓存。
+ **服务端持久化存储介质【必做】**：用户界面配置、订单展示数据、资质信息等核心数据，存储于服务器硬盘、云数据库存储介质，采用MySQL+Redis组合存储，保障界面数据调取速度。
+ **敏感数据存储介质【必做】**：用户手机号、资质证件等界面展示的敏感数据，禁止存储于客户端本地，仅加密存储于服务端专用安全存储介质，界面按需调取且不落地缓存。
+ **备份存储介质【选做】**：界面配置、用户偏好数据定期备份至云存储备份介质，防止数据丢失导致界面异常。

## 4.5 界面基础设计补充
结合用户角色与差异化需求，补充界面基础规范，贴合前文功能优先级：

+ **跨端适配【必做】**：RN框架实现Android、iOS端界面布局统一，无样式错乱、适配断层问题。
+ **老年人/无障碍模式【选做】**：一键切换后界面全局大字、简化布局、高对比度，核心按钮放大，删减非必要入口。
+ **规则提示【必做】**：所有运营规则通过问号图标悬浮提示，界面无隐藏条款、无遮挡提示框。

# 5 接口需求
本章明确软件运行过程中涉及的各类接口规范，分为**软件接口**（内部业务交互、外部系统对接）与**硬件接口**（移动端设备硬件调用）两大类，统一接口请求方式约定：**数据查询/获取类操作采用GET请求，数据新增/删除/修改/提交类操作采用POST请求**，保障接口调用规范、数据流转稳定，适配前端RN、后端Node.js技术栈交互逻辑。

## 5.1 接口通用规则
+ **请求方式**：纯数据查询、列表获取、详情查看接口使用**GET**；用户注册、订单提交、信息修改、资质上传、状态更新等操作使用**POST**。
+ **数据格式**：接口请求/响应统一采用JSON格式，编码为UTF-8，接口返回包含状态码、提示信息、业务数据三部分，异常场景返回标准错误码。
+ **权限校验**：核心业务接口需携带用户令牌（Token）校验身份，区分乘客、车主、管理员三类角色权限，禁止越权调用。

## 5.2 软件接口
软件接口覆盖内部业务全流程与第三方系统对接，按业务模块拆分，明确接口功能与请求方式，标注优先级：

### 5.2.1 用户管理接口【必做】
+ 用户注册接口（POST）：提交手机号、密码、身份类型（乘客/车主）完成账号创建
+ 用户登录接口（POST）：验证账号密码，生成登录Token，返回用户基础信息
+ 用户信息查询接口（GET）：获取个人资料、资质状态、信用评级等信息
+ 用户信息修改接口（POST）：更新个人基础属性、紧急联系人、模式偏好（老年/无障碍）
+ 车主资质上传接口（POST）：提交驾驶证、行驶证图片及信息，发起资质审核
+ 资质状态查询接口（GET）：查看车主资质审核结果与有效期

### 5.2.2 订单与拼车接口【必做】
+ 拼车单发布接口（POST）：乘客提交出发地、目的地、出行时间等信息，生成拼车需求
+ 订单列表查询接口（GET）：获取乘客/车主可匹配订单、历史订单列表
+ 车主筛选乘客接口（GET）：按出行需求、匹配度查询可接单乘客列表
+ 乘客筛选车主接口（GET）：按路线重合度、信用评级查询可接单车主列表
+ 订单接单/取消接口（POST）：车主确认接单、乘客/车主取消订单
+ 订单详情查询接口（GET）：查看订单行程、费用、司乘信息详情

### 5.2.3 安全管控接口【必做（加密）+选做（轨迹监控）】
+ 敏感信息加密接口（POST）：加密传输车主手机号、证件等隐私数据【必做】
+ 轨迹上传接口（POST）：实时上传行车导航轨迹数据【选做】
+ 路线偏移警报接口（POST）：触发异常路线警报，推送通知至相关方【选做】
+ 行程共享接口（POST）：将实时行程同步至紧急联系人【选做】

### 5.2.4 合规运营接口【必做】
+ 地域规则查询接口（GET）：获取当前城市准入政策、接单上限等合规规则
+ 高速费核算接口（GET）：根据行程里程计算高速分摊费用
+ 接单次数统计接口（GET）：查询车主当日已接单量，判断是否超限

### 5.2.5 特色功能接口【选做】
+ 碳积分查询接口（GET）：获取用户拼车减碳积分与成长树状态
+ 积分兑换接口（POST）：提交积分兑换优惠券申请
+ 评价标签提交接口（POST）：上传司乘双方评价与个性化标签

### 5.2.6 外部系统对接接口【必做】
+ 高德地图对接接口（GET）：获取定位、路径规划数据【必做】；路线偏移判定数据【选做】
+ 实名认证对接接口（POST）：提交身份信息，获取第三方核验结果【选做】
+ 多语言切换接口（GET）：获取对应语种界面文本词条【选做】
 
## 5.3 硬件接口
硬件接口用于移动端APP调用设备硬件能力，保障核心功能落地，均为必做项：

### 5.3.1 定位硬件接口
+ GPS定位调用接口：获取设备实时经纬度、地理位置信息，支撑拼车匹配与轨迹监控
+ 定位权限申请接口：请求用户开启GPS定位权限，反馈权限开启状态

### 5.3.2 网络通信接口
+ 移动网络/Wi-Fi调用接口：检测网络连接状态，保障接口数据传输通畅
+ 网络状态监听接口：实时监控网络波动，断网时提示用户，联网后自动重连

### 5.3.3 外设交互接口
+ 摄像头调用接口（选做）：调用设备摄像头上传资质证件照片（核心为图片采集，非行车监控）
+ 音频播放接口：语音播报行程信息、警报提示，适配无障碍/老年人模式
+ 屏幕显示适配接口：根据设备分辨率、模式切换（老年/无障碍）调整界面渲染参数

# 7 数据需求
本章针对软件运行全流程的各类业务数据，明确**输入输出格式、收发频率、准确性精度、数据流量、保存时效**五大核心要求，覆盖用户数据、订单数据、轨迹数据、合规数据、特色功能数据等全类型数据，贴合项目轻量化运营定位，保障数据流转规范、存储合理、调用高效。

## 7.1 输入、输出数据格式
所有数据统一遵循标准化格式，兼顾前端RN渲染、后端Node.js处理、第三方系统对接的兼容性，核心格式要求如下：

+ **基础文本/数值数据【必做】**：采用UTF-8编码，用户信息、订单参数、合规规则等结构化数据，统一用**JSON格式**传输；金额、里程、时间等数值数据，严格遵循界面需求的格式规范（金额保留2位小数、里程保留1位小数、时间为YYYY-MM-DD HH:MM）。
+ **多媒体/证件数据【必做】**：驾驶证、行驶证、头像等图片数据，采用**WebP/JPG压缩格式**，单张图片大小≤5MB；语音播报数据采用MP3格式，适配移动端音频播放。
+ **轨迹/定位数据【必做】**：经纬度、行程轨迹数据采用浮点型数值格式，搭配时间戳封装为JSON数组传输，保证地图SDK正常解析。
+ **特色功能数据【选做】**：碳积分、成长树状态、评价标签等数据，采用整型+字符串组合格式，同步兼容数据库存储与界面可视化渲染。

## 7.2 接收、发送数据的频率
按数据类型与业务场景，分频次管控数据收发节奏，避免频繁请求造成性能压力，具体规则：

+ **实时高频数据【必做】**：行程轨迹、定位数据，**每3秒发送1次**至服务端；路线偏移监测数据，实时触发上传（异常即发）；订单状态变更、消息通知，**即时推送**至对应客户端。
+ **中频交互数据【必做】**：订单筛选、列表查询数据，客户端请求频率限制为**1次/秒**；用户信息、资质状态查询，**按需请求**（无操作不触发）；碳积分、信用评级数据，**每5分钟同步1次**。
+ **低频静态数据【必做】**：地域合规规则、多语言词条、界面配置数据，**每日同步1次**；用户历史订单、评价数据，客户端主动查询时触发收发，无主动查询不传输。
+ **特色功能数据【选做】**：成长树状态、优惠券数据，**每次登录/行程结束后同步1次**，不做高频收发。

## 7.3 数据的准确性和精度
严格管控数据误差，保障业务逻辑正常运行，精度要求贴合性能需求章节标准：

+ **核心业务数据【必做】**：用户身份信息、资质核验结果、订单金额、高速分摊费，**准确率100%**，无缺失、无错漏；金额计算精度保留小数点后2位，无四舍五入误差。
+ **定位轨迹数据【必做】**：GPS定位经纬度误差≤5米，路线重合度计算误差≤5%，轨迹数据无断点、无偏移，保证行程监控准确性。
+ **统计类数据【必做】**：接单次数、订单量、活跃用户数等统计数据，**准确率≥99.9%**；碳积分、减碳值等特色数据，精度取整数，误差≤1%【选做】。
+ **评价标签数据【选做】**：司乘评价、标签数据，存储准确率100%，不丢失、不篡改。

## 7.4 数据流量
基于轻量化运营场景，管控单用户与系统整体数据流量，避免带宽过载：

+ **单用户日均流量【必做】**：普通用户（仅核心拼车功能）日均消耗流量≤50MB；开启行程实时监控的用户，日均流量≤100MB，轨迹数据采用压缩传输降低流量消耗。
+ **单请求流量上限【必做】**：单次接口请求/响应数据包大小≤500KB，单次图片上传≤5MB，禁止超大数据包传输。
+ **系统整体流量【必做】**：项目初期日均总流量≤50GB，峰值时段每秒总流量≤10Mbps，适配服务端带宽配置。
+ **特色功能流量【选做】**：成长树、积分兑换等数据传输，单用户单次流量≤100KB，不占用核心业务带宽。

## 7.5 数据需保持的时间
按数据重要性与合规要求，分级设定数据保存时效，兼顾存储成本与数据追溯：

+ **用户核心数据【必做】**：注册信息、实名认证资质、信用档案，**永久保存**（用户注销后7个工作日内彻底删除）。
+ **订单行程数据【必做】**：已完成/已取消订单、行程轨迹、费用记录，**保存2年**，满足合规追溯与用户查询需求。
+ **临时缓存/日志数据【必做】**：接口缓存、运行日志、定位临时数据，**自动清理时效7天**，避免占用存储资源。
+ **合规管控数据【必做】**：地域规则、接单记录、警报日志，**保存1年**，用于平台运营核查。
+ **特色功能数据【选做】**：碳积分、成长树、评价标签，**与用户账号生命周期一致**，用户注销后同步删除。

# 9 安全保密需求
本章聚焦软件全流程安全管控与隐私保密，针对用户敏感信息、业务数据、系统权限制定刚性规则，与性能需求的安全指标形成互补（性能重效率、本章重管控），杜绝信息泄露、数据篡改、越权操作、恶意攻击等安全风险，保障司乘双方权益与平台合规运营。

## 9.1 用户隐私保密要求
+ **敏感信息加密【必做】**：用户手机号、身份证号、驾驶证、行驶证、紧急联系人信息等核心隐私数据，全程采用加密传输+加密存储，禁止明文展示、明文存储，仅授权人员可脱敏查看。
+ **信息权限管控【必做】**：严格划分乘客、车主、管理员权限边界，禁止越权查看他人隐私；车主联系方式仅订单存续期间对匹配乘客可见，行程结束后自动脱敏隐藏。
+ **隐私授权合规【必做】**：获取定位、存储、电话等设备权限前，需明确告知用户用途，仅获取功能必需权限，禁止私自采集无关隐私数据。

## 9.2 数据安全防护要求
+ **数据防篡改【必做】**：订单数据、行程轨迹、费用金额、资质信息等核心业务数据，采用校验机制防护，杜绝后台篡改、恶意修改，保障数据真实性。
+ **数据备份恢复【必做】**：核心数据每日自动备份，支持故障后快速恢复，避免数据丢失；备份数据同样加密存储，防止泄露。
+ **数据销毁合规【必做】**：用户注销账号后，7个工作日内彻底删除其所有隐私数据与业务数据，不残留、不复用。

## 9.3 系统安全与权限管控
+ **登录安全校验【必做】**：支持账号密码验证，可选短信验证码登录，限制异常登录次数，防止暴力破解账号。
+ **管理员权限管控【必做】**：管理员操作实行分级授权，核心操作（资质审核、数据删除、规则修改）需留痕记录，便于追溯。
+ **恶意行为拦截【必做】**：实时拦截恶意刷单、虚假资质、非法请求等行为，屏蔽异常账号，保障平台秩序。

## 9.4 传输与运行安全
+ **网络传输安全【必做】**：所有接口数据采用HTTPS加密传输，防止网络劫持、数据窃听。
+ **运行漏洞防护【必做】**：开发阶段排查代码漏洞，避免内存泄漏、越权访问等安全隐患；后期定期巡检，修复安全漏洞。

# 11 质量保证
本章聚焦系统全生命周期质量管控，与性能需求形成区分（性能重指标阈值、本章重管控机制与落地要求），围绕可靠性、错误处理、变更维护、可移植性七大核心要点制定标准，实现质量可量化、问题可追溯、迭代可管控。

## 11.1 系统可靠性要求
立足轻量化运营场景，保障系统持续稳定运行，杜绝核心功能失效、大面积故障：

+ **核心功能可用率**：订单、定位、轨迹监控等核心模块可用率≥99%，非核心模块可用率≥97%；
+ **业务连续性**：单个模块故障不影响全局运行，避免单点故障引发系统崩溃；
+ **数据可靠性**：核心数据读写、存储无丢失、无错乱，网络波动场景下数据不丢失。

## 11.2 系统错误监测与隔离要求
+ **实时错误监测**：系统内置监控模块，实时捕捉进程异常、接口超时、数据错误、越权操作等全类型故障，无监测盲区；
+ **分级错误隔离**：监测到故障后自动分级处置，轻微错误仅记录日志，一般错误限制模块访问，严重错误隔离故障模块，保留核心功能运行，杜绝故障扩散；
+ **错误日志追溯**：所有故障自动生成日志，记录故障详情，便于后期排查优化。

## 11.3 系统平均无故障时间（MTBF）要求
+ **核心系统**：平均无故障运行时间（MTBF）≥720小时（30天），保障长期稳定运营；
+ **全系统整体**：平均无故障运行时间≥480小时（20天），满足日常使用稳定性需求。

## 11.4 系统出错重启恢复时限
+ **常规热重启**：进程异常、闪退等常规故障，重启后30秒内完成功能恢复；
+ **强制冷重启**：严重故障冷启动后，5分钟内完成系统复原、数据校验；
+ **服务端重启**：后端服务故障重启，10分钟内恢复正常业务服务。

## 11.5 系统变更的设计反馈机制
+ **变更审批管控**：功能迭代、规则调整、参数优化等所有系统变更，需经审批后方可实施，禁止随意变更；
+ **设计同步更新**：变更实施前，同步更新需求文档、原型、接口文档等设计文件，保证文档与实际系统一致；
+ **变更测试闭环**：变更完成后需开展回归测试，验证无关联故障后方可上线。

## 11.6 系统维护与改进要求
+ **维护范畴包含改进**：系统维护不仅包含故障修复，还涵盖性能优化、体验升级、安全强化、合规迭代等改进性工作；
+ **定期维护巡检**：每周开展基础维护，每月进行系统优化，提前排查潜在隐患；
+ **持续迭代改进**：结合用户反馈与运营数据，定期制定优化方案，持续提升系统质量。

## 11.7 系统可移植性要求
+ **跨端移植**：基于RN框架开发，支持Android、iOS双端无缝移植，无需大规模重构代码；
+ **服务端移植**：后端服务支持跨云厂商、跨操作系统移植，配置适配简单；
+ **数据移植**：核心数据采用标准化格式，支持完整导出导入，迁移无丢失、无错乱。
