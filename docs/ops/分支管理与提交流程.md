# 分支管理、合并与提交流程

> 本文档用于团队在 GitHub 上进行分支管理、合并与提交发布的统一操作说明。具体规范以项目根目录 `CONTRIBUTING.md` 为准。

## 目标

- 保证 `main` 分支始终保持可部署状态
- 降低多人协作冲突概率
- 让 PR 评审与合并过程可追溯、可复用

## 分支类型与使用时机

1. `main/master`：生产环境稳定分支，仅允许通过 PR 合并发布分支/热修复分支进入。
2. `develop`：日常开发汇总分支，所有功能分支最终合并到该分支。
3. `feature/xxx`：新功能开发分支，从 `develop` 拉出，完成后提交 PR 合并到 `develop`。
4. `fix/xxx`：测试环境/集成环境 bug 修复分支，从 `develop` 拉出，完成后合并回 `develop`。
5. `hotfix/xxx`：生产环境紧急修复分支，从 `main` 拉出，完成后合并回 `main` 和 `develop`。

## 最小提交流程（提交前检查）

在执行 `git add` / `git commit` / `git push` 前，建议至少完成以下检查：

- 本地代码是否符合 `CONTRIBUTING.md` 的编码规范与接口约束
- 前端：`npm run lint`；提交前建议 `npm run test:ci`
- 后端：确认接口方法约定与参数校验/响应标准化；提交前建议 `npm test`
- 不提交敏感信息（`.env`、密钥、token 等）

## 提交（Commit）规范

- commit 信息必须遵循 `CONTRIBUTING.md` 的标准结构：`<type>(<scope>): <subject>`
- 单次提交尽量聚焦单一功能/单一 bug/单一优化
- 复杂修改建议补充 body/footers（如关联需求/bug ID、BREAKING CHANGE 标注等）

## 合并（Merge）与 PR 流程

1. 从目标分支拉出工作分支（如 `feature/xxx`、`fix/xxx`、`hotfix/xxx`）。
2. 完成开发后按“功能拆分”提交多次 commit（不要把所有变更打成一坨）。
3. 推送工作分支到远程仓库。
4. 在 GitHub 创建 PR：
   - PR 标题遵循 commit/PR 规范
   - PR 描述写明修改原因、影响范围、测试要点
   - 至少指定 1 名评审人（按团队规则执行）
5. 等待 CI/自动检查与评审意见完成后合并：
   - 处理评审指出的问题后重新提交（或更新 PR）
6. 合并后删除临时分支（`feature/fix/hotfix`），避免分支长期堆积。

## 发布与版本标注（可选）

- 生产环境发布后，在 `main` 分支打版本标签（例如 `vx.x.x`）
- 标签建议与 PR/里程碑对应，便于回滚与追溯

## 验证清单（建议）

- 前端：至少保证关键页面可进入、关键接口调用无方法/路径错误
- 后端：保证服务可启动、接口返回结构符合响应标准化格式
- 联调：确认前后端方法约定（查询 `GET`、CUD `POST`）一致

## Git 命令示例（可复制）

以下示例以仓库根目录为 `E:\carpooling`，工作分支命名使用 `feature/xxx`，目标分支按你们 `CONTRIBUTING.md` 的约定填写（常见为 `develop` 或 `main`）。

### 1) 准备与拉取

```bash
cd E:\carpooling
git switch main
git pull --rebase origin main
```

### 2) 新建功能分支

```bash
git switch -c feature/rn-pages
```

### 3) 提交前检查

```bash
git status
git diff
```

### 4) 分批提交（git add + commit）

```bash
# 批次 A：前端目录结构/路由/配置变更
git add -A frontend
git commit -m "refactor(rn-structure): 对齐前端目录结构与配置命名"

# 批次 B：规范/文档更新
git add CONTRIBUTING.md backend/README.md
git commit -m "docs(node-api): 更新后端分层目录说明以匹配编码规范"
```

### 5) 推送到远程

```bash
git push -u origin feature/rn-pages
```

### 6) 发起 PR（gh 示例）

```bash
gh pr create --base develop --head feature/rn-pages \
  --title "feat(rn-pages): 实现/迁移页面结构" \
  --body "说明修改原因、影响范围、测试要点"
```

### 7) 拉取 PR 更新到本地（可选）

```bash
gh pr checkout
```
