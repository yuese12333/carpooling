## 一、全局约定

### 1.1 命名规范

| 层级 | 约定 | 示例 |
|------|------|------|
| Prisma model | PascalCase | `AuthUser`、`RideOrder` |
| 数据库表名 | snake_case + `@@map` | `auth_users`、`ride_orders` |
| 主键 | `String @id @db.VarChar(64)`（业务主键） | `user_id`、`order_id` |
| 自增主键 | `BigInt @id @default(autoincrement()) @db.UnsignedBigInt` | `EmergencyContact.id` |
| 时间字段 | `created_at`（`@default(now())`）+ `updated_at`（`@updatedAt`） | 所有 model |
| 金额字段 | `Decimal @db.Decimal(10, 2)` | `price`、`total_amount` |
| 外键关联 | `onDelete: Cascade`（跟随主数据删除）或 `SetNull`（主数据删除后保留记录） | 见 §3 |

### 1.2 外键删除策略速查

| 策略 | 含义 | 使用场景 |
|------|------|----------|
| `Cascade` | 主记录删除 → 关联记录全部删除 | 用户删除时其 profile、收藏地点等跟随删除 |
| `SetNull` | 主记录删除 → 关联字段置 NULL，记录保留 | 审计日志、违规记录、订单等需保留历史的场景 |

## 五、业务层实现注意事项

### 5.1 账户与认证

#### 5.1.1 `role` vs `role_type` 的区分

- `AuthUser.role`：**平台权限角色**，值域 `user` / `admin`。控制能否访问管理后台。
- `UserProfile.role_type`：**用户身份类型**，值域 `passenger` / `driver`。标识用户是乘客还是司机。
- `TripParticipant.role`：**行程内角色**，值域 `driver` / `passenger`。标识该用户在某具体行程中的身份。

**查询时必须区分**：判断管理员用 `auth_users.role`，判断司机身份用 `user_profiles.role_type`。

#### 5.1.2 `status` 字段与登录拦截

登录流程必须在密码校验通过后检查 `auth_users.status`：
- `status === 'disabled'` → 拒绝登录，返回 401。
- 不存在该用户或密码错误 → 返回 401。

#### 5.1.3 `RealNameAuth.verify_status` 默认 `pending`

新建实名认证记录时状态为 `pending`，需走审核流程后才能变为 `verified`。与 `DriverCredential` 保持一致。

#### 5.1.4 `UserProfile.rating_avg` 默认 `5.00`

新用户未收到评价时显示满分，收到评价后按均值更新。这是行业常见做法，无需修改。

### 5.2 行程与订单

#### 5.2.1 两条业务线的关系

本项目存在**两条独立的业务线**：

| 业务线 | 流程 | 涉及的表 |
|--------|------|---------|
| **行程发布线** | 司机发布 `Ride` → 乘客加入 `TripParticipant` | `rides`、`trip_participants` |
| **需求匹配线** | 乘客发布 `RideRequest` → 司机接单生成 `RideOrder` | `ride_requests`、`ride_orders` |

两条线在 schema 层**无直接关联**。业务实现时需要注意：

1. `RideOrder` 完成后，应在 DAO/service 层**同步创建或更新** `TripParticipant` 记录（用于行程列表展示）。
2. `TripParticipant.status` 与 `RideOrder.order_status` 需保持一致，建议在订单状态变更时同步更新参与记录。
3. 建议在 service 层文档化这条同步链路，避免两套状态不一致。

#### 5.2.2 `seats_left` 库存校验

- 乘客加入行程时，必须在**事务中**执行：`UPDATE rides SET seats_left = seats_left - 1 WHERE ride_id = ? AND seats_left > 0`
- 如果 `seats_left` 变为 0，同步更新 `ride_status = 'full'`。
- 取消订单时反向恢复：`UPDATE rides SET seats_left = seats_left + 1`。

#### 5.2.3 `RideOrder.request_id` 为 NOT NULL

所有订单必须从 `RideRequest` 产生，不允许脱离需求直接创建订单。创建订单时先校验 `request_id` 对应的 `RideRequest` 是否存在且状态为 `open`。

#### 5.2.4 `Ride` 搜索索引

已建立两个复合索引支持乘客搜索：
- `idx_rides_status_from_depart`(ride_status, from_text(10), depart_at)
- `idx_rides_status_to_depart`(ride_status, to_text(10), depart_at)

查询时**必须包含 `ride_status`** 作为第一条件才能命中索引。

### 5.3 支付与财务

#### 5.3.1 `transaction_id` 唯一约束

`OrderPayment.transaction_id` 为 UNIQUE。第三方支付回调重试时：
- 同一 `transaction_id` 重复回调 → Prisma 会报 UNIQUE 冲突。
- DAO 层应先 `findUnique` 检查是否已存在，存在则直接返回成功（幂等），不存在则 `create`。

#### 5.3.2 `payment_method` 为字符串快照

`OrderPayment.payment_method` 不关联 `PaymentMethod.method_id`，存储的是下单时支付方式名称（如 `"alipay"`）。原因：支付方式可能被删除，但订单需保留历史记录。

#### 5.3.3 `Receipt.issued_at` 无默认值

发票开具时间需由业务层根据实际开票时机显式设置。创建 Receipt 时必须传入 `issued_at`。

#### 5.3.4 `is_default` 多选的应对

`Vehicle`、`PaymentMethod`、`UserLocation` 的 `is_default` 字段在库层面允许多条 `true`。DAO 层在设置新默认值时，**必须先清除该用户的旧默认值**：

```js
// 设置默认车辆时
await prisma.vehicle.updateMany({
  where: { owner_user_id: userId, is_default: true },
  data: { is_default: false },
});
await prisma.vehicle.update({
  where: { vehicle_id: targetId },
  data: { is_default: true },
});
```

### 5.4 安全与合规

#### 5.4.1 `UserViolation` 防重复处罚

表层面无唯一约束（`order_id` 可 NULL，MySQL UNIQUE 对 NULL 宽松）。DAO 层在创建违规记录前应先检查是否已存在同用户、同订单、同类型的违规记录。

#### 5.4.2 `compliance_region_code` FK 约束

`RideOrder.compliance_region_code` 关联 `ComplianceRegionRule.region_code`。创建订单时：
- 若写入不存在的 region_code → 数据库会拒绝。
- 合规规则删除时，该字段自动置 NULL（`onDelete: SetNull`）。

#### 5.4.3 订单合规校验

司机接单时应校验：该司机当日已完成订单数 < `ComplianceRegionRule.daily_driver_order_limit`。查询方法：

```js
const todayOrders = await prisma.rideOrder.count({
  where: {
    driver_user_id: driverId,
    created_at: { gte: startOfDay, lte: endOfDay },
    compliance_region_code: regionCode,
  },
});
```

#### 5.4.4 `trip_locations` 数据膨胀

轨迹点表按 GPS 定时写入，数据量增长快。建议：
- 行程结束后停止上报。
- 定期归档历史数据（如超过 30 天的轨迹点迁移到冷存储）。
- 查询时务必带 `order_id` + `recorded_at` 范围以命中复合索引。

#### 5.4.5 `TripParticipant.joined_at` 无默认值

创建参与记录时必须显式赋值 `joined_at`，通常与 `created_at` 使用同一时间。

### 5.5 评价与社交

#### 5.5.1 `OrderRating` 评分范围

`score` 为 `Int`，无库级范围约束。DAO 层写入前必须校验 `1 <= score <= 5`。

#### 5.5.2 评价完成后更新 `UserProfile.rating_avg`

收到新评价后，需更新被评价人的 `rating_avg`：

```js
const avgResult = await prisma.orderRating.aggregate({
  where: { to_user_id: userId },
  _avg: { score: true },
});
await prisma.userProfile.update({
  where: { user_id: userId },
  data: { rating_avg: Decimal(avgResult._avg.score ?? 5.00) },
});
```

同时更新 `total_completed_orders`。

#### 5.5.3 `TripParticipant.is_rated` 同步

订单互评完成后，需更新该行程所有参与者的 `is_rated = true`，防止重复评价。

#### 5.5.4 `InviteRecord` 唯一约束

`@@unique([invite_code, invitee_user_id])` 防止同一邀请码对同一被邀请人重复产生奖励记录。创建前无需预检查，直接 `create`，若冲突则视为已发放。

#### 5.5.5 `InviteCode` 生成策略

邀请码为 `VARCHAR(20)` 主键，生成时需在应用层保证唯一性。推荐方案：使用 UUID 截取前 20 位或雪花算法，若 UNIQUE 冲突则重新生成。

### 5.6 通知

#### 5.6.1 `Notification.type` 默认 `"info"`

通用通知默认类型为 `info`。创建通知时根据场景选择合适的 type：
- `info`：系统通知、行程提醒
- `success`：订单完成、支付成功
- `warn`：安全预警、信用分下降
- `error`：支付失败、行程异常

### 5.7 碳账户

#### 5.7.1 事务一致性

`CarbonAccount` 与 `CarbonTransaction` 必须在同一事务中更新：

```js
await prisma.$transaction([
  prisma.carbonTransaction.create({ data: { ... } }),
  prisma.carbonAccount.upsert({
    where: { user_id: userId },
    create: { user_id: userId, carbon_points: delta, total_reduced_kg: reducedKg },
    update: { carbon_points: { increment: delta }, total_reduced_kg: { increment: reducedKg } },
  }),
]);
```

### 5.8 审计日志

#### 5.8.1 `AdminAuditLog` 审计人可为 NULL

`admin_user_id` 和 `target_user_id` 为可空字段。管理员账号被删除后，其操作记录保留但关联置 NULL。写入时仍需传入当前管理员 ID。

#### 5.8.2 所有管理操作必须写审计日志

管理后台的每一个写操作（改状态、改角色等）都必须调用 `admin_audit_logs.create()` 记录。

### 5.9 Prisma 连接注意事项

- **单例模式**：使用 `global.__prisma` 避免热重载连接泄漏。
- **事务**：涉及多表写入的操作（如订单创建 + 座位扣减 + 碳积分）必须使用 `prisma.$transaction`。
- **软删除**：`UserLocation` 使用 `is_deleted` 标记而非真实删除。查询时必须加 `where: { is_deleted: false }`。
- **状态字段**：当前所有状态字段使用 `String`（非 Prisma enum），DAO 层写入前需校验值域。后续新增 model 建议使用 `enum`。

### 5.10 日志与监控

#### 5.10.1 日志表数据膨胀

`event_logs`、`error_logs`、`performance_logs` 三张日志表数据增长快，建议：
- 定期归档历史数据（如超过 30 天的数据迁移到冷存储）
- 查询时务必带时间范围条件以命中索引
- 考虑按时间分表或使用时序数据库

#### 5.10.2 敏感信息脱敏

日志表存储时必须脱敏：
- `error_logs.request_data`：移除密码、token 等敏感字段
- `event_logs.event_data`：移除手机号、身份证等 PII

#### 5.10.3 性能日志阈值

`performance_logs.duration_ms` 建议只在超过阈值（如 500ms）时记录，避免日志泛滥。

### 5.11 通知设置

#### 5.11.1 用户注册时自动创建

新用户注册时，DAO 层应自动创建默认 `NotificationSetting` 记录，避免查询时返回空。

#### 5.11.2 设置更新幂等

用户重复提交相同设置时，DAO 层应使用 `upsert` 或先查询后更新，确保幂等。

### 5.12 搜索历史

#### 5.12.1 历史记录去重

同一用户重复搜索相同路线时，建议更新 `created_at` 而非新增记录，或限制每个用户的历史记录数量（如最多 50 条）。

#### 5.12.2 定期清理

建议定期清理超过一定时间的搜索历史（如 90 天）。

### 5.13 协议文档

#### 5.13.1 版本管理

`protocols.version` 用于版本控制。`force_update = true` 时，用户需重新同意协议才能继续使用。

#### 5.13.2 协议类型

`type` 字段为主键，支持的协议类型：
- `user_agreement`：用户协议
- `privacy_policy`：隐私政策
- `driver_agreement`：司机服务协议
