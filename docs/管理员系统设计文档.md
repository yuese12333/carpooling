# 管理员系统设计文档

> 本文档记录管理员系统的完整设计方案，供开发参考。

## 1. 方案选型

采用 **单表扩展方案**：在现有 `auth_users` 表中新增角色与状态字段，不另建 `admin_users` 主表。

选择该方案的原因：

- 与现有登录链路完全兼容，无需改动认证核心逻辑。
- 后端改造量最小，降低引入新 bug 的风险。
- 前端只需新增管理页面，不需要维护两套登录体系。

## 2. 数据库改造

### 2.1 为 `auth_users` 表新增字段

**第一步：更新 `prisma/schema.prisma`**

在 `AuthUser` model 中新增两个字段（`@@map("auth_users")` 对应的 model）：

```prisma
model AuthUser {
  // ... 已有字段 ...
  role      String   @default("user")   @db.VarChar(20)
  status    String   @default("active") @db.VarChar(20)
}
```

**第二步：执行 DDL（生产/测试环境直接执行 SQL）**

```sql
ALTER TABLE auth_users
  ADD COLUMN role   VARCHAR(20) NOT NULL DEFAULT 'user'   COMMENT '角色: user / admin',
  ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'active' COMMENT '状态: active / disabled';

CREATE INDEX idx_auth_users_role   ON auth_users(role);
CREATE INDEX idx_auth_users_status ON auth_users(status);
```

**第三步：重新生成 Prisma Client**

```bash
npx prisma generate
```

> **注意**：项目使用 Prisma ORM，仅执行原始 SQL 不会让 Prisma Client 感知新字段。必须同步更新 `schema.prisma` 并重新执行 `prisma generate`，否则 `prisma.auth_users.findFirst()` 等查询访问不到 `role` 和 `status` 字段。

### 2.2 初始化管理员账号

先通过 **`POST /api/auth/register`** 创建一个普通账号（见 [`用户注册接口联调文档.md`](./用户注册接口联调文档.md)），再执行以下 SQL 将其提升为管理员（密码沿用 bcrypt，无需修改）：

```sql
UPDATE auth_users
SET role = 'admin'
WHERE phone = '13900000000';
```

### 2.3 审计日志表（建议实现，答辩展示效果好）

```sql
CREATE TABLE IF NOT EXISTS admin_audit_logs (
  id             BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  admin_user_id  VARCHAR(64)     NOT NULL COMMENT '操作管理员 ID',
  action         VARCHAR(50)     NOT NULL COMMENT '操作类型',
  target_user_id VARCHAR(64)     NOT NULL COMMENT '被操作用户 ID',
  detail         JSON            NULL     COMMENT '操作详情',
  created_at     TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_admin_audit_admin_user_id  (admin_user_id),
  KEY idx_admin_audit_target_user_id (target_user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

同样需要在 `schema.prisma` 中声明对应 model，并重新执行 `npx prisma generate`：

```prisma
model AdminAuditLog {
  id             BigInt   @id @default(autoincrement()) @db.UnsignedBigInt
  admin_user_id  String   @db.VarChar(64)
  action         String   @db.VarChar(50)
  target_user_id String   @db.VarChar(64)
  detail         Json?
  created_at     DateTime @default(now())

  @@index([admin_user_id], name: "idx_admin_audit_admin_user_id")
  @@index([target_user_id], name: "idx_admin_audit_target_user_id")
  @@map("admin_audit_logs")
}
```

## 3. 权限模型

**角色（role）**

| 值 | 含义 |
|---|---|
| `user` | 普通用户 |
| `admin` | 管理员 |

**账号状态（status）**

| 值 | 含义 |
|---|---|
| `active` | 正常，可登录 |
| `disabled` | 已禁用，禁止登录 |

登录成功后，JWT token payload 中需携带 `role` 字段：

```json
{
  "userId": "u_xxx",
  "phone": "13900000000",
  "role": "admin"
}
```

**改造位置**：`backend/src/service/auth-service.js` 的 `loginByPassword()` 函数。

现有的 `jwtUtils.generateToken(userId, phone, extraPayload, expireTime)` 已支持透传额外字段，只需在调用时将 `role` 放入 `extraPayload`：

```javascript
// auth-service.js — loginByPassword 中，密码校验通过后
// 1. 先检查账号状态（见第 6 节规则 1）
if (user.status === 'disabled') {
  throw Object.assign(new Error('账号已被禁用，无法登录'), { statusCode: 401 });
}

// 2. 生成 token 时透传 role
const token = jwtUtils.generateToken(
  user.user_id,
  user.phone,
  { role: user.role },   // ← 新增
  expireTime
);
```

社交登录（`loginBySocial`）如需支持管理员，同理在 `extraPayload` 中传入 `role`。

后端使用两个中间件实现权限控制：

- `auth-middleware`：校验 token 有效性，解析当前用户信息，将解析结果挂载到 `req.user`。
- `admin-auth-middleware`：在 `auth-middleware` 之后执行，校验 `req.user.role === 'admin'`。

## 4. 后端接口设计

> 遵循项目规范：只使用 `GET` / `POST`，不使用 `PUT` / `PATCH` / `DELETE`。

### 4.1 查询用户列表

- **方法与路径**：`GET /api/admin/users/list`
- **鉴权**：`auth-middleware` + `admin-auth-middleware`
- **查询参数**：

| 参数 | 是否必填 | 说明 |
|---|---|---|
| `page` | 否 | 页码，默认 1 |
| `pageSize` | 否 | 每页条数，默认 10，最大 50 |
| `phone` | 否 | 手机号，模糊匹配 |
| `userName` | 否 | 昵称，模糊匹配 |
| `role` | 否 | 角色筛选 |
| `status` | 否 | 状态筛选 |

### 4.2 更新用户状态

- **方法与路径**：`POST /api/admin/users/update-status`
- **鉴权**：`auth-middleware` + `admin-auth-middleware`
- **请求体**：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `targetUserId` | 是 | 被操作用户的 ID |
| `status` | 是 | `active` 或 `disabled` |

### 4.3 更新用户角色

- **方法与路径**：`POST /api/admin/users/update-role`
- **鉴权**：`auth-middleware` + `admin-auth-middleware`
- **请求体**：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `targetUserId` | 是 | 被操作用户的 ID |
| `role` | 是 | `user` 或 `admin` |

### 4.4 关于删除用户

本项目**不提供真实删除接口**。管理员通过 `update-status` 将用户设为 `disabled` 即视为封禁，数据保留。

## 5. 后端目录结构

新增文件按如下位置放置，与项目现有分层保持一致：

```
backend/src/
├── router/
│   └── admin-router.js          ← 新建
├── controller/
│   └── admin-controller.js      ← 新建
├── service/
│   └── admin-service.js         ← 新建
├── dao/
│   └── admin-dao.js             ← 新建
└── middleware/
    ├── schema-init-guard.js     ← 已有，不改动
    ├── auth-middleware.js        ← 新建（基于 jwt-utils.js 实现 token 解析）
    └── admin-auth-middleware.js  ← 新建（校验 role === 'admin'）
```

调用顺序严格遵循：`router → controller → service → dao`，禁止跨层调用。

在 `backend/src/index.js` 中注册新路由：

```javascript
const adminRouter = require('./router/admin-router');
app.use('/api/admin', adminRouter);
```

### 5.1 auth-middleware 实现要点

`auth-middleware.js` 是项目目前**缺失**的基础中间件，需要新建，核心逻辑：

```javascript
const jwtUtils = require('../utils/jwt-utils');
const { buildFailureResponse, createRequestId } = require('../utils/response');

function authMiddleware(req, res, next) {
  const requestId = req.headers['x-request-id'] || createRequestId();
  const authHeader = req.headers['authorization'];
  const token = authHeader?.startsWith('Bearer ') ? authHeader.slice(7) : null;

  if (!token) {
    return res.status(401).json(buildFailureResponse(401, '未登录或 token 缺失', null, requestId));
  }

  try {
    const payload = jwtUtils.verifyToken(token);
    req.user = payload;   // { userId, phone, role, ... }
    next();
  } catch (err) {
    return res.status(401).json(buildFailureResponse(401, 'token 无效或已过期', null, requestId));
  }
}

module.exports = authMiddleware;
```

### 5.2 admin-auth-middleware 实现要点

```javascript
const { buildFailureResponse, createRequestId } = require('../utils/response');

function adminAuthMiddleware(req, res, next) {
  const requestId = req.headers['x-request-id'] || createRequestId();

  if (req.user?.role !== 'admin') {
    return res.status(403).json(buildFailureResponse(403, '无管理员权限', null, requestId));
  }
  next();
}

module.exports = adminAuthMiddleware;
```

## 6. 关键业务规则

以下规则必须全部实现：

1. **禁用用户不可登录**：`auth-service.js` 的 `loginByPassword()` 在密码校验通过后、生成 token 前，检查 `user.status === 'disabled'`，若为禁用则抛出 401 错误。
2. **管理员不能操作自己**：不能将自己降级为 `user`，也不能将自己设为 `disabled`。在 `admin-service.js` 中校验 `targetUserId !== req.user.userId`。
3. **所有管理操作写入审计日志**：调用 `admin_audit_logs` 记录操作人、操作类型、被操作对象。
4. **日志脱敏**：禁止在日志中打印明文密码、token、完整手机号。使用项目现有的 `maskSensitive()` 工具函数（`backend/src/utils/mask-utils.js`）。
5. **统一响应结构**：所有管理接口返回格式如下，使用项目现有的 `buildSuccessResponse` / `buildFailureResponse`（`backend/src/utils/response.js`）：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "requestId": "RN-xxxx"
}
```

补充说明：`requestId` 允许前端透传（如 `RN-...`），后端在缺失时可兜底生成（如 `BE...`），前后端无需强制同一前缀。

## 7. 前端页面设计

### 7.1 页面路由

| 路径 | 说明 |
|---|---|
| `/admin/login` | 管理员登录（可复用普通登录页逻辑，见 7.3） |
| `/admin/users` | 用户管理列表页（核心页面） |
| `/admin/no-permission` | 无权限提示页 |

### 7.2 用户管理列表页功能

- **列表字段**：手机号、昵称、角色、状态、注册时间（对应 DB 字段 `created_at`）
- **条件筛选**：手机号、昵称、角色、状态
- **分页**：页码 + 每页条数
- **行操作**：
  - 启用 / 禁用（调用 `update-status`）
  - 设为管理员 / 普通用户（调用 `update-role`）

### 7.3 前端权限守卫

1. 登录使用现有登录接口，登录成功后将 token 存入全局状态（`useEnvStore`）。
2. 登录成功后，前端从 token payload 中解析 `role` 字段（可使用 `jwt-decode` 或 Base64 解析中段）。
3. 若 `role !== 'admin'`，前端**不跳转后台**，而是显示无权限提示或跳转 `/admin/no-permission`。后端不拒绝非 admin 用户登录，role 检查发生在前端路由层，以及后端的 `admin-auth-middleware`。
4. 进入任意 `/admin/*` 路由前，路由守卫校验 token 有效性与 role，无效则跳转 `/admin/login`。

> **注意**：`/admin/login` 复用普通登录页时，需在登录成功回调中额外执行 role 检查逻辑，而不是直接跳转首页。

### 7.4 requestId 规范

管理员页面遵循项目现有的 requestId 链路规范：
- 页面层通过 `useMemo(() => generateRequestId(), [])` 生成 `requestId`（`RN-` 前缀）。
- 通过 props 注入 hook，hook 再传递给 API 函数。
- `request.ts` 拦截器自动将 `currentRequestId` 注入 `X-Request-Id` 请求头。
- **禁止**在 API 函数内部手动设置 `headers: { 'X-Request-Id': requestId }`，会绕过拦截器导致 trace 链路断裂。

## 8. 开发实施顺序

建议按以下顺序推进：

1. 更新 `prisma/schema.prisma`，新增 `role`、`status` 字段及 `AdminAuditLog` model；执行 DDL SQL；执行 `npx prisma generate`。
2. 修改 `auth-service.js` 的 `loginByPassword()`：查询 `role` 和 `status`，禁用账号返回 401，token 中透传 `role`。
3. 新建 `middleware/auth-middleware.js` 和 `middleware/admin-auth-middleware.js`。
4. 实现后端 3 个管理接口：`list`、`update-status`、`update-role`（dao → service → controller → router 顺序实现）。
5. 在 `index.js` 注册 `adminRouter`。
6. 接入审计日志写入逻辑（在 `admin-service.js` 的操作方法中写入 `admin_audit_logs`）。
7. 实现前端 `/admin/users` 页面与路由权限守卫。
8. 联调与回归测试。
