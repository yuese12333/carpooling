# Expo React Native项目 安卓原生工程版本控制说明文档

> [!IMPORTANT]
>
> **文档目的**：规范Expo React Native项目中安卓原生文件夹（`android/`）的版本管控规则，统一团队原生工程基线，避免因成员各自生成、修改原生文件导致的编译冲突、配置不一致、功能异常等问题，保障协作效率与项目稳定性。
> 
> **适用范围**：本项目全体开发成员，仅针对Expo Prebuild工作流下的安卓原生工程，纯托管工作流可参考简化执行。
> 
> **核心前提**：安卓原生文件夹采用 **“核心配置入Git、动态产物全忽略”** 的部分管控模式，不整体忽略、不全员随意生成，全程保持基线统一。

# 一、核心管控原则

1. **专人负责制**：指定1名核心开发作为安卓原生工程维护人，全权负责原生文件夹的生成、更新、提交，其余成员禁止私自操作预生成命令。

2. **基线唯一制**：远程仓库的安卓原生核心配置为团队唯一基线，所有成员本地原生工程必须与远程基线完全一致，禁止私自修改、覆盖核心配置。

3. **精准忽略制**：仅忽略原生工程内动态生成的编译产物、本地环境配置、IDE缓存文件，核心配置文件100%纳入Git版本控制。

4. **禁止自行生成**：普通成员**严禁**单独执行 `npx expo prebuild --platform android` 命令，杜绝本地原生文件与远程基线冲突。

# 二、Git版本控制范围界定

## 2.1 必须纳入Git跟踪的核心配置文件

以下文件为安卓原生工程核心基线，必须全部提交至远程仓库，保障团队配置完全统一，缺失会导致项目编译失败、第三方SDK（地图、导航等）功能异常：

- 根级配置：`android/build.gradle`、`android/settings.gradle`

- 应用级配置：`android/app/build.gradle`、`android/app/proguard-rules.pro`

- 清单文件：`android/app/src/main/AndroidManifest.xml`（权限、SDK配置、签名相关核心文件）

- 资源文件：`android/app/src/main/res/` 目录下所有静态资源（图标、布局、字符串配置等）

- 原生依赖配置：Expo自动生成的原生模块关联配置文件

## 2.2 必须加入.gitignore的忽略文件

以下文件为动态产物、本地专属配置或缓存文件，严禁纳入Git，避免仓库体积膨胀、环境冲突，需在项目根目录`.gitignore`中精准配置如下规则：

```gitignore
# ------------------------------
# Expo 安卓原生工程 忽略规则
# ------------------------------
# 编译构建产物
android/app/build/
android/build/

# Gradle 相关缓存
android/.gradle/
android/**/*.iml

# 本地环境专属配置
android/local.properties
android/gradle.properties

# IDE 与系统缓存
android/.idea/
android/captures/
android/**/.DS_Store

# Expo 动态生成常量
android/app/src/main/assets/expo-constants/

# 原生调试与临时文件
android/app/src/main/jni/
android/app/build.gradle.kts.bak

```

# 三、团队分工与职责

## 3.1 原生工程维护人（指定1人）

- 负责首次执行预生成命令，生成标准安卓原生工程基线，提交至远程开发分支；

- 仅维护人可执行预生成、清理重构、原生配置修改操作，修改后同步提交远程并通知全员；

- 处理原生工程相关Git冲突，同步团队成员的原生配置需求，保障基线合规；

- 管控第三方SDK（地图、导航等）原生配置，确保SHA1、权限、密钥配置统一。

## 3.2 普通团队成员

- 严禁私自执行`expo prebuild`、`expo eject`等修改原生工程的命令；

- 全程同步远程仓库的原生工程基线，不私自修改、删除、新增`android/`目录下任何核心配置文件；

- 本地调试如需临时修改原生配置，仅做本地测试，严禁提交至远程，测试完成后恢复基线；

- 遇到原生工程相关问题，统一反馈至维护人处理，不自行排查修改。

# 四、标准化操作流程

## 4.1 首次基线搭建（仅维护人执行）

1. 确认本地Expo SDK、Node.js版本与团队约定一致，拉取最新项目代码；

2. 执行预生成命令，生成标准安卓原生工程：`npx expo prebuild --platform android`；

3. 核对.gitignore规则，确保仅核心配置文件被Git跟踪，动态产物已忽略；

4. 提交原生工程至远程，提交信息遵循团队Git规范：`chore(android): 初始化安卓原生工程统一基线`；

5. 推送至开发分支，通知全体成员拉取同步。

## 4.2 成员日常同步流程（全员执行）

1. 拉取远程最新代码，同步维护人提交的原生工程基线：`git pull origin develop`；

2. 安装项目依赖，保障依赖版本与团队一致：`npm install`；

3. 直接运行安卓项目，无需任何预生成操作：`npx expo run:android`；

4. 运行后生成的编译产物、缓存文件，因.gitignore规则，不会被Git跟踪，无需手动处理。

**关键提醒**：若本地提前生成过`android/`文件夹，需先删除该目录，再执行拉取操作，彻底避免本地文件与远程基线冲突。

## 4.3 原生工程后续更新（仅维护人执行）

1. 场景触发：修改`app.json`配置、升级Expo SDK、新增第三方原生SDK、调整权限；

2. 执行清理重构命令，确保配置完全同步：`npx expo prebuild --platform android --clean`；

3. 核对核心配置文件（尤其是AndroidManifest.xml、build.gradle），确认自定义配置未被覆盖；

4. 本地编译测试无误后，提交更新，提交信息标注修改内容：`chore(android): 更新地图SDK权限与SHA1配置`；

5. 推送远程，通知全员拉取最新基线。

# 五、冲突处理规范

1. 普通成员禁止处理原生工程相关Git冲突，若拉取代码时提示`android/`目录冲突，立即放弃本地修改，强制同步远程基线：`git reset --hard origin/develop`；

2. 维护人处理冲突时，优先保留远程基线配置，核对本地修改内容，仅合并合规的核心配置变更；

3. 冲突解决后，本地编译验证通过，再提交推送，避免将冲突代码推入远程。

# 六、禁忌行为与红线

- ❌ 严禁普通成员自行执行`npx expo prebuild`生成安卓原生工程；

- ❌ 严禁私自修改`android/`目录下核心配置文件并提交远程；

- ❌ 严禁将.gitignore忽略的编译产物、缓存文件、本地配置提交至远程；

- ❌ 严禁多人同时修改原生工程配置，所有需求统一由维护人处理；

- ❌ 严禁随意删除远程`android/`目录或核心配置文件。

# 七、常见问题排查

1. **问题1**：本地运行提示原生文件缺失 → 解决方案：删除本地`android/`文件夹，重新拉取远程基线；

2. **问题2**：Git提示安卓文件冲突 → 解决方案：普通成员强制重置同步，维护人核对基线后处理；

3. **问题3**：第三方SDK（地图、导航）失效 → 解决方案：核对远程基线的AndroidManifest.xml配置，确保SHA1、权限与维护人提交的一致，不使用本地生成的配置。
