## 二、项目概述与后端业务设计目标

计划书目标包括：实现行程发布、搜索、预订、评价完整闭环；前后端分离；支持 Mock 模式。本人负责的 home/rides/trips/payments/notifications 等 **16 个路由模块中的 6 类业务服务**，是计划书「行程模块」「个人中心（支付/通知）」「首页模块」在后端的直接实现。

后端业务设计目标主要包括：第一，**分层清晰**，严格遵循 Router → Controller → Service → DAO，Service 层承载核心业务规则，不跨层调用；第二，**接口规范统一**，查询用 GET、写操作用 POST，响应体统一为 `{ code, message, data, requestId }`；第三，**业务完整**，覆盖接口汇总清单中首页、找车/发布、我的行程、支付、通知等阶段四核心模块；第四，**可测试**，Service 层编写单元测试，覆盖正常路径与异常分支；第五，**可联调**，字段命名与前端 Mock 数据、API 封装保持一致，降低联调成本。

## 三、后端技术选型与总体架构

### 3.1 技术选型

| **技术** | **版本/说明** | **在本模块中的作用** |
| --- | --- | --- |
| Node.js + Express | 18+ / 4.x | HTTP 服务与路由挂载 |
| Prisma ORM | 6.x | 通过 DAO 层访问 MySQL，事务与关联查询 |

### 3.2 分层架构与模块划分

本人实现的业务模块按路由前缀划分如下：

| **路由前缀** | **Service 文件** | **主要能力** |
| --- | --- | --- |
| `/api/home` | `home-service.js` | 用户信息、推荐行程、系统统计、未读通知状态 |
| `/api/rides` | `ride-service.js` | 行程搜索、发布、详情、预订、路线规划、发布权限 |
| `/api/trips` | `trip-service.js` | 我的行程列表、取消、评价、联系人、取消原因、行程模板 |
| `/api/payments` | `payments-service.js` | 余额、支付记录、支付方式、收据、月度统计 |
| `/api/notifications` | `notification-service.js` | 通知列表、清除通知、创建通知 |
| `/api/help` | `help-service.js` | 帮助分类与常见问题 |

各模块均配备对应的 `router`、`controller`、`dao` 文件，Controller 只做参数接收与响应封装，业务判断集中在 Service 层。

### 3.3 Service 层与 DAO 层的协作关系

以「寻找车程」为例：`ride-controller` 接收 query 参数 → `ride-service.searchRides` 校验分页与筛选条件 → 调用 `ride-dao.searchRides` 执行 Prisma 查询 → Service 将数据库 snake_case 字段映射为前端 camelCase 响应 → Controller 通过 `response.js` 返回标准 JSON。这种结构使我在编写单元测试时可以 Mock DAO，独立验证业务分支，而不依赖真实数据库连接。

## 四、主要业务模块详细设计与实现

### 4.1 行程搜索与发布模块（ride-service）

行程模块是拼车系统的核心业务。`searchRides` 支持出发地/目的地文本、出发日期、价格区间、最少座位数、排序方式等筛选，并将司机信息、车辆信息一并组装返回；登录用户搜索时异步写入 `search_history` 表，便于后续推荐优化。`publishRide` 在发布前校验用户是否绑定车辆、座位数是否合法、出发时间是否有效；若无默认车辆且未指定 `vehicleId`，返回业务错误提示。`bookRide` 实现座位扣减与订单创建，需处理「行程不存在」「座位不足」「司机不能预订自己行程」等分支。`getRideDetail` 返回行程详情及司机档案，供前端详情页展示。

### 4.2 我的行程模块（trip-service）

我的行程模块连接「预订之后」的用户体验。`getTripList` 按状态（即将出行/已完成/已取消）分页查询 `trip_participants` 关联数据。`cancelTrip` 校验行程归属与当前状态，已取消行程不可重复取消；取消时尝试触发退款占位逻辑并向相关用户发送通知。`rateTrip` 校验行程是否已完成、是否重复评价，区分乘客评司机与司机评乘客。`getCancelReasons` 从数据库读取取消原因配置，并按乘客/司机类型过滤。该模块单元测试共 33 条，分支覆盖率达 66.66%，是本人测试中覆盖最充分的模块。

### 4.3 首页与支付、通知模块

`home-service` 聚合用户资料、推荐开放行程、平台统计数据和未读消息状态，为前端首页一次加载多个接口提供数据支撑。`payments-service` 实现余额查询、支付历史分页、支付方式列表、月度统计及取消行程时的 `issueRefund` 退款占位逻辑。`notification-service` 提供通知列表、按 ID 批量清除及业务侧创建通知的能力，在行程取消等场景中被 trip-service 调用。

### 4.4 编码规范遵循情况

开发过程中严格遵循团队 `CONTRIBUTING.md` 规范：文件命名采用 kebab-case（如 `ride-service.js`）；函数以小驼峰动词开头；每条业务日志通过 `logger.info/error` 输出，携带 `module`、`operate`、`requestId`，敏感参数经脱敏处理；异常通过 `error.statusCode` 区分 400/403/404 等业务错误，由 Controller 统一捕获并返回标准响应体。

## 八、遇到的问题与解决方案

**问题一：Service 与 DAO 职责边界不清。** 初期曾在 Service 中直接写 Prisma 查询，导致文件过长且难以测试。解决方式是将所有 SQL/Prisma 操作下沉到 DAO，Service 只保留业务规则与数据组装，单元测试 Mock DAO 即可覆盖分支。

**问题二：取消行程与退款、通知的联动。** 取消行程后需尝试退款并通知对方，涉及 payments-service 与 notification-service 跨模块调用。解决方式是在 trip-service 中以 try/catch 包裹退款与通知逻辑，失败时记录 Warn 日志但不阻断取消主流程，保证核心操作可用。

**问题三：前后端字段命名不一致。** 数据库为 snake_case，前端 TypeScript 为 camelCase。解决方式是在 Service 层统一做字段映射，Controller 不做转换，前端 API 层收到的始终是 camelCase，减少联调时的字段对不上问题。

**问题四：预订接口联调依赖测试数据。** 搜索接口可返回 seed 数据，但预订需座位充足的开放行程。解决方式是在 seed 脚本中预置多条不同状态的行程，并在联调文档中说明测试账号与行程 ID。

## 九、实习经历、感想与收获

本次软件工程课程设计让我第一次完整参与了后端业务模块从设计到联调的全过程。与实验课中写单个接口不同，拼车系统的 Service 层需要同时考虑数据一致性（座位扣减）、权限控制（只能取消自己的行程）、跨模块协作（取消触发退款与通知）以及与前端的字段契约，这让我体会到业务逻辑层在架构中的核心地位。

在编码规范方面，团队统一的日志格式、响应结构和 Git 提交规范，起初觉得繁琐，但在联调阶段证明非常有效——通过 requestId 可以快速定位是前端参数问题还是 Service 业务拦截，而标准化的 `{ code, message, data }` 使前端错误处理逻辑可以复用。

在测试方面，编写 trip-service 33 条单元测试的过程，迫使我梳理每一条业务分支：什么状态下可以取消、什么状态下可以评价、重复操作如何拒绝。测试不是答辩的「凑数」，而是帮助发现逻辑漏洞的有效手段。ride-service 和 trip-service 的分支覆盖率分别达到 30% 和 66%，让我理解了「路径测试」与「语句测试」在工程实践中的意义。

在团队协作方面，我深刻感受到后端业务开发处于「承上启下」的位置：向上要理解产品需求与接口契约，向下要依赖数据库同学提供的表结构，横向要与前端同学对齐字段与状态码。只有主动沟通、及时更新接口文档，才能避免联调阶段的大量返工。

总体而言，本次实习使我掌握了 Express 分层架构、Prisma DAO 模式、Jest 单元测试与 Supertest 集成测试等后端工程化技能，也培养了从业务规则出发设计 Service、而非「对着页面堆接口」的工程思维。今后将继续加强 Controller 层与 Router 层的自动化测试覆盖，并深入学习事务处理与并发场景下的座位扣减方案。
