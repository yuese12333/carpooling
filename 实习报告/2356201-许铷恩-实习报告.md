## 二、项目概述与日志模块设计目标

手机私家车拼车软件系统面向私家车主与乘客之间的拼车出行场景，系统需要支持用户注册登录、车辆管理、行程发布与搜索、订单管理、支付与通知、地图导航等功能。日志模块作为系统的横向基础设施，需要为所有业务模块提供统一的日志记录能力，覆盖前端 React Native 与后端 Node.js 两端。

日志模块的设计目标主要包括以下几个方面：第一，规范统一，前后端日志格式完全一致，字段顺序固定，便于全链路日志关联；第二，接入简单，封装为开箱即用的工具函数，组员通过一行 import 即可调用；第三，安全合规，内置敏感数据脱敏工具，禁止手机号、密码、Token 等敏感信息明文出现在日志中；第四，全链路可追溯，通过唯一请求 ID 串联前后端日志，出问题时可从前端操作一路追查到后端数据库；第五，环境感知，开发环境输出所有级别日志，生产环境自动关闭调试日志，无需手动切换。

## 三、日志与调试关系调研

在开发过程中，当程序出现问题时，常见的排查方式是随手在代码中添加 console.log 输出变量值，查看后再删除。这种方式虽然简便，但存在几个明显问题：格式不统一导致信息难以对比；上线后忘记删除会泄露敏感数据；多人协作时日志信息各异，无法建立全局视图；出现问题时无法快速定位是前端、接口层还是数据层的异常。

规范化日志与上述临时调试方式的本质区别在于：日志是程序运行时自动产生的结构化记录，相当于代码的「行车记录仪」，不依赖开发者手动添加，不受主观遗漏影响。出现问题时，开发者可以通过日志快速还原操作上下文、定位异常模块、追查调用链路，从而将调试从「猜测」变为「查证」。

全链路可追溯是本项目日志设计的核心目标之一。以用户登录失败为例：前端在发起请求时生成唯一请求 ID，记录「用户点击登录」的 INFO 日志，并将请求 ID 通过请求头传给后端；后端在接收请求、校验参数、查询数据库、返回结果的每个关键节点，都使用相同的请求 ID 记录日志。一旦出现问题，开发者只需在控制台或日志系统中搜索该请求 ID，即可看到完整的前后端调用链，而不需要在大量日志中逐条比对时间戳。

## 四、日志规范设计

### 4.1 日志级别划分

参照业界通用日志分级标准，结合本项目前后端技术栈，制定五级日志规范如下：

| **级别** | **调用方法** | **使用场景** | **生产环境** |
| --- | --- | --- | --- |
| DEBUG | logger.debug() | 开发调试，查变量值、追踪局部逻辑，生产环境完全禁用 | 自动关闭 |
| INFO | logger.info() | 正常业务流程，关键操作成功，接口请求/响应记录 | 显示 |
| WARN | logger.warn() | 非阻断性异常，参数不规范但不影响主流程，需关注 | 显示 |
| ERROR | logger.error() | 功能异常，接口失败，需定位修复，必须包含完整异常上下文 | 显示 |
| FATAL | logger.fatal() | 系统崩溃，核心功能不可用，需立即触发告警处理 | 显示 |

### 4.2 日志必填字段规范

每条日志必须包含固定核心字段，缺失字段视为违规，字段顺序固定不得调整。

| **字段名** | **适用端** | **说明** |
| --- | --- | --- |
| 级别 | 前端 + 后端 | DEBUG / INFO / WARN / ERROR / FATAL |
| 时间 | 前端 + 后端 | 精确到秒，格式：2026-03-21 14:30:25 |
| requestId | 前端 + 后端 | 全链路唯一请求 ID，前端生成后通过请求头 x-request-id 传给后端 |
| module | 前端 + 后端 | 所属模块，使用 kebab-case，如 user-login、order-create |
| device | 仅前端 | 设备标识，自动读取 iOS/Android 系统版本，如 iOS-16 |
| serverNode | 仅后端 | 服务节点标识，自动读取主机名，如 node-server-01 |
| operate | 前端 + 后端 | 操作描述，如「用户登录」「创建订单」 |
| params | 前端 + 后端 | 核心业务参数，敏感数据必须经脱敏工具处理后再传入 |
| result | 前端 + 后端 | 执行结果，仅 INFO/WARN 级别必填 |
| error / errorType | 前端 + 后端 | 错误描述与错误类型，仅 ERROR/FATAL 级别必填 |

### 4.3 敏感数据脱敏规范

日志中禁止出现用户密码、完整手机号、Token、身份证号等隐私数据。脱敏规则如下：手机号中间四位替换为星号，如 138****1234；密码、Token、密钥等字段一律替换为 ******；嵌套对象中的敏感字段递归脱敏处理。所有敏感字段必须先调用内置 maskSensitive() 工具函数处理后再传入 params 字段，禁止手动拼接脱敏字符串。

### 4.4 日志格式示例

合规 INFO 日志示例：

```
[INFO] [2026-03-21 14:30:25] [requestId: RN202603211430001] [module: user-login] [device: iOS-16] [operate: 用户登录] [params: phone:138****1234] [result: 登录成功，token生成完成]
```

合规 ERROR 日志示例：

```
[ERROR] [2026-03-21 14:31:10] [requestId: RN202603211430001] [module: order-create] [serverNode: node-01] [operate: 创建订单] [params: driverId:1001] [error: 司机当前不可接单] [errorType: 业务参数异常]
```

违规日志示例（禁止）：仅打印「登录失败」，无 requestId、无模块、无错误原因；明文打印用户手机号 13812341234、密码 123456；同一操作连续重复打印多条日志。

## 五、通用日志工具开发

### 5.1 前端日志工具（React Native）

文件路径：`frontend/src/utils/logger.ts`。该工具面向 React Native（Expo）前端开发，使用 ES Module 语法（import/export），适配 TypeScript 类型体系。

核心功能包括：支持五级日志输出，通过 React Native 内置变量 __DEV__ 判断环境，开发环境显示全部级别，生产环境自动关闭 DEBUG；自动调用 Platform API 读取设备操作系统与版本号，填充 device 字段；内置 generateRequestId() 函数，生成格式为 RN202603211430001 的唯一请求 ID；内置 maskSensitive() 函数，通过关键词匹配自动识别并脱敏 phone、password、token、idCard 等敏感字段，支持嵌套对象递归处理；输出格式严格按照规范字段顺序拼接，与后端格式完全一致。

调用方式示例：

```typescript
import logger, { maskSensitive, generateRequestId } from '@/utils/logger';

logger.info({ module: 'user-login', operate: '用户登录', params: maskSensitive({ phone }), result: '登录成功', requestId: rid });
```

### 5.2 后端日志工具（Node.js）

文件路径：`backend/src/utils/logger.js`。该工具面向 Node.js 后端开发，使用 CommonJS 语法（require/module.exports），与项目后端规范保持一致。

核心功能与前端版本完全对等，主要差异在于：使用 process.env.NODE_ENV 判断环境，生产环境关闭 DEBUG；device 字段替换为 serverNode，通过 Node.js 内置 os.hostname() 自动获取服务器主机名；模块导出包含 logger 对象、maskSensitive 函数与 generateRequestId 函数，供各层代码按需引入。

后端调用方式示例：

```javascript
const { logger, maskSensitive } = require('../utils/logger');

logger.error({ module: 'user-login', operate: '用户登录', params: maskSensitive({ phone }), error: error.message, errorType: '接口请求异常', requestId });
```

## 十、遇到的问题与解决方案

第一个问题是前端与后端的模块语法不同。React Native 使用 ES Module 的 import/export 语法，Node.js 使用 CommonJS require/module.exports 语法。虽然两端的日志工具逻辑完全相同，但必须分别实现，不能共用同一文件。解决方式是明确区分两个文件的定位，在命名、注释和文档中都说明各自适用的环境，避免组员引入时混淆。

第二个问题是敏感数据脱敏的覆盖范围。最初版本只对 phone 和 password 做了处理，但实际业务中还有 token、idCard、bankCard、refreshToken 等字段同样属于敏感数据。解决方式是整理完整的敏感关键词列表，通过字段名关键词匹配实现自动识别，并支持嵌套对象递归脱敏，减少组员手动判断的负担。

第三个问题是导航页面在 iOS 的兼容性。高德原生 SDK 只在 Android 端支持，iOS 端调用 AmapLocation 模块会返回 undefined。解决方式是在功能入口处通过 Platform.OS 判断平台，非 Android 端提前显示友好提示，不触发原生模块调用，保证应用不会因模块缺失而崩溃。

## 十一、实习经历、感想与收获

通过这次的课程设计，我对日志在软件工程项目中的重要性有了更深理解。日志不是简单的调试手段，而是系统可观测性的核心基础设施。在多人协作的项目中，如果每个人用不同格式、不同方式输出调试信息，出现问题时很难能够快速定位到是哪一层的异常。而制定统一的日志规范并将其封装成可直接调用的工具，能够把「日志」从个人习惯转变成团队共识，从临时手段变成持续能力。

在工具开发方面，我第一次完整地实现了一个面向团队使用的基础设施组件，需要同时考虑功能完整性、调用简便性和边界情况处理。例如脱敏工具需要支持嵌套对象、需要覆盖足够多的敏感字段类型、需要在不影响正常字段的前提下精准替换。这些细节让我体会到，做好一个「小工具」并不比实现一个业务功能容易，它需要对使用场景有足够的预判。

在代码审查方面，我第一次系统地以审查者身份检查他人代码，重点关注日志规范的执行情况；在**收尾前端测试**中，通过 `test:ci` 批量回归 Hook 与快照，保证页面迭代不破坏已有行为。

总体来看，这次的课程设计不仅让我提升了工程实践能力，也让我更加理解软件工程中「规范化」「文档化」「可维护」这些原则的实际意义。在未来的项目中，我会更重视基础设施建设和团队协作工具的质量，而不只是关注单一功能的实现。
