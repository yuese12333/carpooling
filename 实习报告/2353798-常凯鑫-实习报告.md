## 二、项目概述与数据库设计目标
手机私家车拼车软件系统面向私家车主与乘客之间的拼车出行场景，系统需要支持用户注册登录、实名认证、车辆管理、常用地点管理、行程发布与搜索、行程参与记录、支付凭证、消息通知、帮助中心、安全风控、评价与信用等功能。数据库作为系统的核心基础设施，需要支撑这些业务模块的数据存储、查询、统计与后续扩展。

数据库设计目标主要包括以下几个方面：第一，结构完整，覆盖计划书第四节全部 P0/P1 模块（认证、行程、个人中心、管理后台等）；第二，关系清晰，通过外键和唯一约束表达用户、车辆、行程、订单、支付等实体之间的业务关系；第三，命名统一，遵守项目 Prisma 开发规范，保证 schema.prisma 与 migrations 成为表结构的唯一真相；第四，便于后端接口开发，后端 DAO 可以直接通过 Prisma Client 访问对应 model，减少手写 SQL 带来的维护成本和安全风险；第五，具备可扩展性，为管理后台审计、评价体系、碳积分、安全审计等功能预留合理字段。计划书非功能需求中「密码加密存储」在库表体现为仅存 `password_hash`；「JWT 认证」依赖 `auth_users` 与 role/status 字段支撑管理后台。

## 三、数据库总体设计方案

### 3.1 技术选型
本项目后端数据库采用 MySQL 作为关系型数据库，使用 Prisma ORM 作为数据库访问与迁移管理工具。MySQL 适合保存拼车系统中的结构化业务数据，例如用户、车辆、行程、订单、支付记录等；Prisma 则提供 schema.prisma 统一建模、Prisma Client 类型化访问、Prisma Migrate 版本化迁移等能力。相比在业务代码中直接执行 CREATE TABLE 或 ALTER TABLE，Prisma 的迁移机制能够让表结构变更进入 Git 版本管理，便于团队审查、回滚和部署。

在实际开发中，我遵循 docs/backend/Prisma开发规范.md 中“表结构唯一真相”的原则：所有 MySQL 表的创建、变更、索引和外键只能通过 backend/prisma/schema.prisma 与 backend/prisma/migrations/ 完成。原先项目中遗留的 schema-dao.js 包含运行时建表 SQL，这种方式不利于版本管理和生产部署，因此我将其中需要保留的历史核心表逐步迁入 Prisma 模型，并补充接口清单中阶段四业务所需的新表。

### 3.2 模块划分
根据业务边界，我将数据库划分为认证与用户资料、实名认证与车辆、地点、行程发布与参与、历史订单与安全、支付与凭证、通知与帮助、邀请与用户成长、评价违规与碳积分、管理后台审计等模块。每个模块既保持相对独立，又通过 user_id、ride_id、order_id、vehicle_id 等业务主键形成关联。

| 业务模块 | 主要数据表 | 设计说明 |
| --- | --- | --- |
| 用户与认证 | auth_users、user_profiles、real_name_auths、user_badges | 保存登录账号、用户资料、实名认证状态、个人成就等基础数据 |
| 车辆与资质 | vehicles、driver_credentials | 保存车辆信息、默认车辆、认证状态、驾驶证与行驶证资料 |
| 地点与出行 | user_locations、rides、trip_participants | 支持常用地点、行程发布、行程搜索、我的行程列表 |
| 订单与安全 | ride_requests、ride_orders、trip_locations、safety_alerts | 支持历史订单、路径记录、偏航检测、安全提醒与合规限制 |
| 支付与通知 | payment_methods、order_payments、receipts、notifications | 支持支付方式、支付记录、电子凭证和消息通知 |
| 运营与扩展 | invite_codes、invite_records、share_events、help_categories、help_questions | 支持邀请好友、分享统计、帮助中心内容维护 |
| 信用与低碳 | order_ratings、user_violations、carbon_accounts、carbon_transactions | 支持评价、违规记录、信用与碳积分体系 |
| 管理后台 | admin_audit_logs | 记录管理员角色与状态调整等关键操作 |


### 3.3 命名与字段规范
为了保证团队协作时可读性一致，我在数据库设计中统一采用以下规则：Prisma model 使用 PascalCase，例如 AuthUser、RideOrder、PaymentMethod；数据库表名使用 snake_case，并通过 @@map 映射到实际表名，例如 auth_users、ride_orders；业务主键采用 String 类型并映射为 VARCHAR(64)，与已有 auth_users.user_id 保持一致；时间字段统一使用 created_at 与 updated_at，分别表示创建时间和更新时间；金额字段使用 Decimal 并映射为 DECIMAL(10,2)，避免浮点数精度问题；对于前端展示中变化较快、结构可能扩展的字段，例如标签、站点列表、额外配置等，使用 JSON 字段保存。

## 四、详细数据库设计与实现

### 4.1 用户认证与资料模块
用户认证模块以 auth_users 为核心表，保存手机号、密码哈希、昵称、头像、登录时间、登录设备信息、角色 role 与账号状态 status。该表既服务普通用户登录，也服务管理后台权限控制。为了保护安全，数据库只保存 password_hash，不保存明文密码。手机号设置唯一约束，防止重复注册。role 和 status 字段配合管理后台实现管理员与普通用户区分、账号启用与禁用控制。

user_profiles 表与 auth_users 形成一对一关系，用于保存用户业务资料，例如出行角色、真实姓名、性别、年龄、信用分、完成订单数、平均评分、累计节省金额和用户等级等。将认证账号与业务资料拆分，可以减少登录查询的数据负担，也使资料扩展更加灵活。real_name_auths 表保存实名认证信息，其中证件号采用哈希和脱敏展示字段分别保存，避免敏感信息直接暴露。user_badges 表记录用户获得的成就勋章，用于个人中心展示。

### 4.2 车辆与资质模块
车辆模块主要由 vehicles 和 driver_credentials 两张表组成。vehicles 表保存车辆 ID、车主用户 ID、车牌号、品牌、型号、颜色、座位数、是否默认车辆、禁烟/空调等服务标签、车辆状态与认证状态。车牌号设置唯一约束，避免同一车辆被重复绑定。driver_credentials 表保存驾驶证号、行驶证号、证件图片地址、审核状态、驳回原因和审核时间等信息，用于车辆认证与司机资质管理。

车辆表与用户表通过 owner_user_id 建立外键关联，删除用户时车辆信息级联删除；行程表 rides 通过 vehicle_id 关联车辆，车辆删除时行程中的 vehicle_id 设置为空，以保留历史行程数据。这样的约束策略既保证了当前数据的一致性，又避免历史订单因车辆解绑而被误删。

### 4.3 地点与行程发布模块
user_locations 表用于保存用户常用地点，包括地点类型（home/work/other）、标签、地址、经纬度、是否默认、是否删除等字段。该设计对应接口清单中的 /api/locations 模块，支持常用出发地和目的地的新增、编辑、删除与搜索。

rides 表是阶段四核心业务中的行程发布表，保存司机、车辆、起点、终点、经纬度、出发时间、总座位数、剩余座位数、价格、途经点、备注、标签、周期规则和行程状态等信息。该表直接支撑 /api/rides/recommend、/api/rides/search、/api/rides、/api/rides/publish-permission 等接口。为了提升搜索效率，我为 ride_status 与 depart_at 设计联合索引，用于按状态和出发时间筛选开放行程。

trip_participants 表用于保存用户参与某个行程的记录，相当于“我的行程”列表的数据基础。它包含 trip_id、ride_id、user_id、角色 passenger/driver、状态 upcoming/completed/cancelled、预订座位数、是否已评价、取消原因等字段。该表通过 ride_id 与 user_id 建立唯一约束，避免同一用户重复参与同一行程。

### 4.4 订单、安全与合规模块
为了兼容旧系统设计和后续订单扩展，我保留并迁入了 ride_requests、ride_orders、trip_locations、safety_alerts 等历史核心表。ride_requests 表表示乘客发起的出行需求，ride_orders 表表示司机与乘客之间形成的订单关系。ride_orders 保存乘客、司机、车辆、订单状态、路线重合率、合规区域、取消原因、开始时间与完成时间等字段。

trip_locations 表保存订单过程中的定位轨迹，包含经纬度、是否偏航、记录时间等字段；safety_alerts 表记录安全提醒，例如偏航、超时、异常停留等事件。这样的设计使系统不仅能完成普通拼车订单，还能为后续安全风控、路线回放、异常报警和合规审查提供数据基础。

### 4.5 支付、通知与帮助模块
支付模块包括 payment_methods、order_payments 和 receipts。payment_methods 保存用户绑定的微信、支付宝、银行卡或余额等支付方式，并记录是否默认；order_payments 保存订单支付金额、路费、乘客应付、司机收入、支付状态、支付方式、交易流水号等字段；receipts 保存电子凭证信息，包括商户、凭证状态、路线起终点和开具时间。

notifications 表用于消息通知，保存通知类别、标题、内容、展示类型、是否已读和读取时间，支持首页未读状态、通知列表和清空通知等接口。help_categories 与 help_questions 用于帮助中心内容，支持按分类展示问题、搜索热门问题，并为后续后台管理帮助内容留下扩展空间。

### 4.6 邀请、评价、违规与碳积分模块
邀请模块由 invite_codes、invite_records 和 share_events 组成。invite_codes 为每个用户生成邀请码，invite_records 记录邀请成功与奖励发放状态，share_events 记录用户分享到微信、QQ 等平台的行为。评价模块由 order_ratings 表实现，保存订单评价双方、评分、评论和标签。违规模块由 user_violations 表保存违规类型、扣分、罚款金额和描述。碳积分模块由 carbon_accounts 与 carbon_transactions 组成，记录用户低碳积分余额、累计减排量和积分流水。

## 六、数据库编码规范与实现过程
在实现过程中，我首先阅读接口汇总清单，梳理前端和后端未来需要落地的业务接口，再根据接口所需数据设计数据库实体。例如 /api/users/me 需要用户资料、评分和累计节省金额，因此需要 user_profiles；/api/vehicles 需要车辆列表、默认车辆和认证状态，因此需要 vehicles；/api/trips 需要用户参与的行程记录，因此需要 trip_participants；/api/payments/history 和 /api/receipts/:id 需要支付记录和凭证，因此需要 order_payments 与 receipts。

随后，我将旧 schema-dao.js 中运行时建表的历史表迁入 Prisma，包括 user_profiles、vehicles、ride_requests、ride_orders、trip_locations、safety_alerts、order_payments、order_ratings、carbon_accounts 等。迁移时我保留了原有表结构中的关键业务字段，同时根据新的前端页面和接口清单补充了 user_locations、rides、trip_participants、payment_methods、notifications、help_categories、help_questions、invite_codes 等新表。

最终实现包括修改 backend/prisma/schema.prisma，新增完整 model、字段、索引、外键和反向关联；新增 backend/prisma/migrations/20260526000000_add_core_business_schema/migration.sql，记录 28 张新增业务表的建表 SQL；并执行 Prisma 格式化、Client 生成和 schema 校验，确保后端接口同学可以直接基于 Prisma Client 调用 prisma.vehicle、prisma.ride、prisma.paymentMethod 等模型。

## 八、运行集成与团队协作
数据库设计完成后，后端接口开发可以按照 Router、Controller、Service、DAO 的分层结构，通过 Prisma Client 访问数据库。例如车辆接口可以使用 prisma.vehicle 查询当前用户车辆列表，行程搜索接口可以使用 prisma.ride 按状态、时间和路线筛选开放行程，我的行程接口可以使用 prisma.tripParticipant 查询用户作为乘客或司机参与的记录。

在协作过程中，我重点保证数据库字段与前端 Mock 数据、接口清单中的响应结构保持一致。例如车辆模块既需要品牌、型号、车牌、颜色、座位数，也需要默认车辆和认证状态；支付模块既需要支付方式列表，也需要支付历史和凭证详情；通知模块需要已读状态和分类筛选。这些字段在数据库层面提前设计，可以减少接口同学后续开发时反复改表的成本。

同时，我在文档中明确 Prisma 是唯一建表入口，避免后端接口在 DAO 或 Service 中继续写 CREATE TABLE 或 ALTER TABLE。这样能够保证数据库结构变更可追踪、可审查、可部署，也让团队在多人协作时减少“本地有表、线上没表”或“代码和数据库结构不一致”的问题。

## 九、遇到的问题与解决方案
第一个问题是旧项目中存在 schema-dao.js 运行时建表逻辑。该方式虽然在早期开发中方便，但不符合正式项目的数据库迁移规范。我通过阅读 Prisma 开发规范，将其中需要保留的历史核心表迁入 schema.prisma，并把后续建表统一交给 Prisma Migrate 管理。

第二个问题是接口清单中的业务模块较多，如果简单按页面堆字段，后续维护会比较混乱。因此我先按业务领域划分模块，再设计实体关系。例如 rides 表负责行程发布，trip_participants 表负责用户参与记录，ride_orders 表负责历史订单与安全合规模块，三者虽然都与“行程”有关，但职责不同，拆分后更利于接口查询和后续扩展。

第三个问题是敏感数据的处理。实名认证、登录密码、手机号、支付信息都涉及隐私和安全。因此我在数据库层面避免保存明文密码，实名认证证件号拆分为哈希和脱敏展示字段，支付方式只保存展示摘要，不保存完整银行卡号或第三方支付敏感凭证。

第四个问题是表关系和删除策略。拼车系统中既有当前业务数据，也有历史记录。对于用户资料、常用地点等强从属数据，我采用级联删除；对于历史订单中关联车辆的字段，我采用 SET NULL 策略，以避免车辆删除后历史订单也被误删。这体现了数据库设计中数据完整性和历史可追溯之间的平衡。

## 十一、实习经历、感想与收获
通过本次课程设计，我对数据库在软件工程项目中的基础作用有了更深理解。数据库设计并不是简单地把页面上的字段保存下来，而是要从业务关系、数据生命周期、约束规则、查询效率、安全性和后续扩展等多个角度进行综合考虑。尤其在拼车系统中，用户、车辆、行程、订单、支付、安全和评价之间存在大量关联，如果前期表结构设计不清晰，后续接口开发和联调会反复受阻。

在实践过程中，我学习并应用了 Prisma ORM 的 schema 建模和迁移机制，理解了为什么正式项目不能依赖运行时自动建表，也不能在 DAO 中随意写 DDL。通过将 schema.prisma 和 migrations 作为表结构唯一真相，数据库变更能够像代码一样被版本管理和审查，这比手工改表更加可靠。

本次实习也让我体会到团队协作的重要性。数据库设计需要对接前端页面需求，也要服务后端接口实现，因此必须同时阅读接口文档、前端 Mock 数据和后端规范。只有把各模块的数据边界提前梳理清楚，后端同学才能更顺利地开发 API，前端同学也能更稳定地完成联调展示。

总体来看，我完成了手机私家车拼车软件系统后端数据库部分的完整搭建工作，包括表结构设计、Prisma 模型编写、迁移文件创建、字段规范和验证流程。通过这次实习，我不仅提升了数据库建模能力，也更加理解软件工程中规范化开发、版本管理、团队协作和可维护设计的重要意义。
