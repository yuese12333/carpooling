## 二、项目概述与前端设计目标

项目名称为 **Carpooling 校园拼车平台**。随着共享经济发展，拼车成为经济、环保的出行方式；本系统面向校园用户，需实现注册登录、行程发布与搜索、在线预订、评价与管理后台，并支持高德地图定位路线与 Mock 模式并行开发（见 `docs/project/项目计划书.md`）。前端作为用户交互入口，需支撑计划书中全部 P0/P1 界面能力落地。

前端设计目标主要包括以下几个方面：第一，功能完整，覆盖从认证到核心业务再到个人资料管理的完整闭环，确保每个页面均配有独立样式、业务Hook和API文件，避免逻辑堆叠；第二，结构清晰，按要求将api、hooks、pages、router、store、utils等目录分层，明确各层职责，便于团队并行开发和后续维护；第三，界面一致，通过全局样式令牌统一颜色、间距、圆角与阴影，保持视觉风格清爽可信赖；第四，可联调可追踪，请求层统一处理baseURL、请求ID、错误响应与Mock模式，确保页面能在真实接口链路中稳定运行，降低联调成本。

## 三、前端技术选型与总体架构

### 3.1 技术选型与运行环境

本项目移动端前端采用React Native与Expo技术路线，允许跨平台适配，使团队可以把更多精力放在业务页面、接口联调和移动端交互逻辑上。页面代码主要使用TypeScript编写。通过为API、页面状态和表单数据定义类型，保持前端代码的可维护性和可读性。

在状态管理方面，项目创建了env-store、auth-context、mock-data等文件，用于保存环境切换、认证信息和模拟数据。请求层则通过request.ts、api-response.ts、sync-request-id.ts等工具文件统一处理接口调用和requestId同步。

### 3.2 前端目录结构与分层设计

项目前端结构为典型的“页面容器+业务Hook+接口服务+工具函数+全局状态”架构。页面层可以保持相对简洁，复杂的业务逻辑被下沉到Hook或工具文件中，接口调用被集中到api目录中，路由路径则统一由router/paths.ts管理。

| **目录****/层级** | **在项目中的作用** |
| --- | --- |
| api | 封装页面所需接口，屏蔽底层请求实现，使页面只关注业务参数和返回结果 |
| hooks | 承载表单状态、页面加载、筛选排序、提交校验等业务逻辑，实现UI与逻辑解耦 |
| pages | 组织具体页面、局部组件和页面样式，是用户界面的主要实现位置 |
| router | 集中维护路由常量，避免页面跳转路径硬编码，降低路由调整时的维护成本 |
| store | 维护认证状态、运行环境、Mock数据和部分跨页面共享状态 |
| utils | 提供请求、日志、校验、导航、地理位置和requestId同步等基础能力 |


在页面内部，项目也采用了进一步拆分。这种拆分使页面的结构与功能语义更接近，也更方便后续复用和单独修改。

### 3.3 页面、Hook、API、Style 的协作关系

在实际开发中，一个完整前端模块通常由四类文件共同构成：页面文件负责渲染界面和组合组件，style文件负责定义该页面的样式，Hook文件负责管理业务状态和事件处理，API文件负责向后端发起请求。例如“寻找车程”模块可以由find-ride.tsx、find-ride.style.ts、use-find-ride-form.ts、find-ride-api.ts和若干局部组件共同组成。页面组件调用Hook获取搜索条件、筛选状态和行程列表，再把数据传递给ride-card等组件展示；当用户点击搜索、筛选或排序时，Hook调用API文件获取数据并更新页面状态。

这种结构的核心价值在于降低耦合。如果将请求、表单校验、状态更新和样式全部写在页面组件中，页面会迅速变得臃肿，并且难以定位错误。拆分后，前端开发者可以先用Mock数据完成UI，再在API层对接后端；也可以在不改变UI的情况下替换接口路径或请求参数；当页面样式需要调整时，不必改动业务Hook。对于多人协作项目而言，这种边界尤其重要，方便维护页面、接口或样式，减少冲突。

## 四、主要页面与功能模块设计实现

### 4.1 登录、注册与找回密码模块

认证模块是用户进入系统的第一入口，也是前端与后端联调最早遇到完整流程的模块。登录页面需要支持手机号与密码输入、字段校验、接口请求、登录成功后的路由跳转，以及错误提示。注册模块采用分步骤表单设计。分步注册的好处是降低用户一次性填写大量信息的压力，也便于在每一步进行局部校验。找回密码模块则包含手机号输入、验证码校验、重设密码和成功提示等步骤，整体设计与注册模块类似。

在认证模块中，我对前端开发的一个重要体会是：登录注册看似简单，但一旦进入真实工程，就会涉及大量边界情况。例如手机号格式不合法时应在前端直接提示，避免无效请求；密码长度和复杂度规则应与后端保持一致；请求失败时应区分网络异常、参数错误和账号密码错误；登录成功后应同步认证状态并跳转到首页；调试时应通过requestId追踪一次完整登录请求，而不是仅靠console.log临时打印；前端不应直接记录密码，而应传递哈希处理后的结果以保证安全等。

### 4.2 首页与快捷操作模块

首页是用户进入系统后的业务入口，需要在有限的手机屏幕中组织核心功能。首页开发要把用户最常用的任务路径压缩到较少步骤。例如乘客希望尽快搜索合适车程，就应能从首页直接进入寻找车程；车主希望发布行程，则应能快速进入提供车程页面；用户想查看历史或即将出行的订单，则应能进入我的行程；导航功能合并后，也需要在快捷操作区提供清晰入口。

### 4.3 寻找车程模块

寻找车程是乘客侧的核心功能。该页面需要支持输入出发地和目的地、筛选时间、查看可用行程、按条件排序并进入行程详情或参与流程。在实现中，寻找车程模块的关键是处理列表状态和筛选状态。前端需要在没有接口数据时使用Mock数据维持页面展示，在接口可用后切换到真实请求；需要在搜索条件为空、结果为空、请求失败和加载中等状态下给出不同反馈；也需要保证卡片信息的表达足够直观，例如起终点、出发时间、剩余座位、价格、司机/车辆信息和标签。对移动端用户而言，卡片内容过多会造成阅读负担，过少又不足以做出选择，因此信息层级设计非常重要。

### 4.4 提供车程模块

提供车程是车主侧的核心功能。该页面相比寻找车程更偏向复杂表单，需要填写路线、出发时间、可提供座位数、价格、备注和服务偏好等内容。提供车程模块对前端表单设计提出了更高要求。首先，出发地与目的地应当清晰区分，并支持常用地点或城市选择；其次，日期时间输入适配移动端控件，避免用户手动输入复杂日期格式；再次，座位数和价格需要进行合法性校验，防止出现负数、空值或超出车辆座位数的情况；最后，备注信息保持可选，不让非必要字段阻碍发布流程。

### 4.5 我的行程、个人中心与资料管理模块

我的行程模块负责展示用户参与或发布的行程记录。该页面区分即将出行、已完成、已取消等状态，并提供空状态提示。

个人中心是系统信息密度最高的区域之一，承担资料展示、菜单导航和二级功能入口。其中包含常用地点、车辆管理、车辆编辑、实名认证、车辆认证详情、支付方式、支付历史、消息通知、帮助中心、邀请好友、支付凭证等页面。个人中心模块的难点在于功能多、入口多、状态多。实名认证状态会影响是否展示认证入口或认证结果；车辆认证状态会影响车辆卡片和编辑入口；支付方式和支付历史涉及不同数据列表；消息通知涉及已读未读；帮助中心则涉及分类与搜索。前端需要尽量把这些状态组织成可理解的菜单和卡片，而不是把所有功能都堆在同一级列表中。

## 五、全局样式体系与界面一致性设计

### 5.1 Design Tokens 设计

本项目包含全局样式体系文件，通过Design Tokens的方式将颜色、间距、圆角、尺寸、阴影和布局混入统一抽象出来。基础调色板PALETTE定义了原子颜色；业务颜色令牌则进一步把这些颜色映射为更细致的语义化变量。这样做的好处是页面开发者在使用颜色时不必关心具体十六进制值，而是根据语义选择“主色”“错误色”“提示色”“边框色”等。同时，通过其他抽象，页面在不同平台上能保持接近的视觉层级，同时减少重复代码。

| **样式体系** | **主要内容** | **设计意义** |
| --- | --- | --- |
| COLORS | primary、bgLight、textMain、textSecondary、error、warning、status等语义颜色 | 统一页面配色，避免每个页面独立使用零散颜色值 |
| SPACING | xs=4、sm=8、md=16、lg=24、xl=32、xxl=40 | 形成统一的间距节奏，使卡片、表单和列表更加规整 |
| RADIUS | xs、sm、md、lg、xl、xxl、full、card | 统一按钮、卡片、标签、输入框的圆角风格 |
| SIZES | navbarHeight、touchTarget、statusBarHeight、buttonHeight、inputHeight | 保证移动端触控区域和常用组件尺寸一致 |
| SHADOWS | xs、sm、lg、primary，并区分iOS与Android | 通过平台适配实现卡片层级和按钮强调效果 |
| LAYOUT_MIXINS | rowCenter、rowBetween、center、mainCard、navbar、badgeBase等 | 抽取常用布局，减少重复样式并提升代码可读性 |


### 5.2 公共组件与页面风格统一

项目先后手动实现了card、avatar、badge等多种公共组件。这些组件的意义在于把高频UI元素从具体页面中抽离出来，使不同页面都能使用相近的按钮、卡片、标签和弹层风格。公共组件越稳定，页面开发就越容易保持一致，也越不容易出现同一类按钮在不同页面中高度、圆角、边框完全不同的问题。

## 六、接口配置、请求封装与前后端联调

前端与后端之间的接口对齐是本项目开发中的重要部分。前端按模块拆分出api部分，使页面调用接口时可以直接引入对应业务服务，而不需要在页面中拼接URL或处理底层请求细节。

请求封装的重点之一是requestId链路。前端每次发起请求时，都需要保证请求头或参数中的requestId能被后端识别，并在日志中保持一致，这样当某个页面请求失败时，团队才能通过同一个ID追踪前端操作、网络请求和后端处理。

接口联调还涉及Mock模式的管理。在后端接口尚未完成或联调环境不稳定时，前端需要先依靠工具模拟接口返回，保证页面开发不会完全停滞。但Mock只能用于过渡，不能长期替代真实接口。因此开发过程中需要明确哪些数据来自Mock，哪些数据已经接入真实API，并在后续提交中逐步修正Mock模式判断和模拟数据类型。

在路由配置方面，项目通过paths.ts定义统一路径管理常量。路由硬编码是前端项目中非常常见的问题：如果每个页面直接写字符串路径，当目录结构调整时很容易遗漏修改，导致点击按钮无响应或跳转错误。统一路由常量可以降低这一风险，也使页面跳转关系更加可查。

## 九、测试、问题定位与解决方案

### 9.1 页面功能测试

前端测试主要围绕页面是否能正常进入、表单是否能正确输入与校验、按钮是否能触发预期操作、列表是否能正确展示数据、空状态和异常状态是否有提示、路由跳转是否正确等方面展开。对于登录注册模块，需要测试手机号格式、密码长度、验证码输入、提交按钮禁用与启用、登录成功跳转和登录失败提示；对于寻找车程模块，需要测试搜索条件、筛选标签、排序面板、列表卡片与无结果状态；对于提供车程模块，需要测试路线、时间、座位、价格和备注等表单项；对于个人中心，需要测试各菜单入口和认证状态显示。

移动端前端还需要特别关注不同屏幕尺寸和键盘弹出后的交互表现。例如在ScrollView或列表页面中，键盘、输入框和按钮点击之间会发生冲突。类似问题在Web页面中不一定明显，但在React Native移动端非常常见，必须通过实际运行和点击测试才能发现。

### 9.2 接口与日志联调问题

在联调过程中，一个常见问题是请求链路不连续。前端页面发起请求、请求层添加requestId、后端接收requestId、日志输出requestId，这几个环节只要有一处遗漏，就会导致追踪断链。解决思路是把requestId的读取与同步尽量放到统一请求层和全局状态中，避免让每个页面手动生成或传递。

另一个问题是日志过多或日志位置不合理。开发阶段为了排查问题，容易在组件中添加大量临时输出，但如果不加控制，会造成控制台信息泛滥，也可能泄露敏感数据。项目采用全局logger和日志规范后，可以把关键操作、请求结果、异常信息以结构化方式记录，同时避免在UI组件中随意使用console.log。对前端来说，日志不只是后端关心的内容，它同样能帮助定位按钮点击、接口调用、页面加载和状态更新的问题。

### 9.3 样式与平台适配问题

样式问题通常表现为页面看似能运行，但视觉不统一或某些平台显示异常。项目曾修正页面主题和样式逻辑、删除badge组件强制黑色边框、修复SafeAreaView导入方式、添加RN-paper Provider嵌套等。这些问题说明React Native页面不仅要考虑JSX结构，还要考虑第三方组件库的上下文要求、iOS与Android的安全区差异、阴影表现差异以及组件默认样式。

解决这类问题的核心方法是减少页面级随意样式，尽量复用全局tokens、公共组件和布局混入。比如同一类卡片统一使用Card，同一类导航栏统一使用navbar，按钮和输入框高度统一使用SIZES。这样即使后期需要调整主题，也能在全局样式文件中集中修改，而不是在几十个页面中逐个查找。

## 十、实习经历、感想与收获

本次软件工程课程设计让我完整经历了从需求理解、模块分工、前端设计、接口封装、联调测试到后期修复的全过程。与平时独立完成作业不同，课程设计中的每个功能都是相互联动的：登录页需对接认证接口、保存状态、跳转首页并记录日志；找车程页需支持搜索、筛选、Mock与接口切换；个人中心牵涉实名、车辆、支付等多个页面。这让我认识到前端开发不仅仅在于把设计的页面做出来，还需要考虑如何将功能纳入完整业务流程。

前端直接面向用户，既需技术实现也需设计判断。移动端屏幕小、输入成本高，必须让任务路径短、反馈及时、信息清晰。拼车系统的核心是帮助乘客找到车程、车主发布车程、用户管理资料与订单，围绕这些任务组织页面比追求视觉复杂更重要。通过分层，明确各层职责，使项目在功能增加后仍可维护。全局样式体系通过统一令牌管理颜色、间距、圆角，避免了重复定义和手动修改的麻烦，让设计规范落地为代码。

在前后端协作中，我体会到不能只关心Mock数据下的表现，还需确保请求字段与后端一致、错误处理正确等处理，统一请求层、路由常量和日志格式，虽会多花时间，但能大幅节省联调排查成本。

总而言之，这次课程设计让我从“写一个页面”过渡到“参与一个项目”，不仅提升了技术能力，更理解了分工协作、规范化开发和工程化思维的意义。前端作为连接用户与后端系统的桥梁，既需关注体验，也需具备架构意识，才能保证项目长期可维护。
