## 二、总体设计方案

### 2.1 项目背景与目标

本项目面向校园用户，开发一款移动端拼车应用，实现「注册登录 → 发布/搜索行程 → 在线预订 → 支付结算 → 行程评价」的业务闭环，并配套管理后台与 Mock 演示模式，满足课程答辩与团队并行开发需求。

### 2.2 总体架构

系统采用**前后端分离 + 四层架构**：

```
客户端（React Native + Expo）
        │  HTTPS / JSON
        ▼
API 网关（Nginx 反向代理、HTTPS 终结）
        │
        ▼
应用服务（Node.js + Express，16 个业务路由模块）
        │
        ▼
数据层（MySQL 主存储 + Redis 缓存 + 本地文件 uploads）
```

**设计原则**：

1. **前后端分离**：前端专注 UI 与交互，后端专注业务与数据，通过 RESTful 风格 API 通信。  
2. **模块化**：认证、首页、行程、支付、通知、管理后台等模块独立路由，低耦合。  
3. **可扩展**：Prisma ORM 管理 39 张表，迁移脚本纳入 Git，便于 schema 演进。  
4. **安全与可追溯**：JWT 鉴权、密码哈希存储、管理员审计日志、全链路 `requestId` 日志追踪。

### 2.3 技术选型

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| 前端 | React Native + Expo | 0.81 / 54 | 一套代码覆盖 iOS/Android，Expo 降低原生配置成本 |
| 前端路由 | Expo Router | 6 | 文件系统路由，与页面目录结构一致 |
| 前端状态 | Zustand | 5 | 轻量，适合 Mock 模式与环境切换 |
| 后端 | Node.js + Express | 18+ / 4.21 | 与前端同语言，团队上手快 |
| ORM | Prisma | 6.19 | 类型安全、迁移可版本化 |
| 数据库 | MySQL | 8.0 | 事务完善，适合订单类业务 |
| 缓存 | Redis | 7 | 短信验证码、频率限制 |
| 认证 | JWT | 9 | 无状态，适合移动端 |
| 地图 | 高德 Android SDK | 6.0 | 国内定位与 POI 搜索准确 |
| 部署 | Nginx + PM2 | — | 测试环境稳定运行 |

### 2.4 功能模块划分

系统共划分 **16 个后端路由模块**，对应前端主要业务页面：

- **认证** `/api/auth`：密码登录、注册、找回密码、Token 刷新  
- **短信** `/api/sms`：注册场景阿里云发码/验码  
- **用户** `/api/users`：当前用户、实名状态、邀请与分享  
- **首页** `/api/home`：用户信息、推荐行程、统计、未读消息  
- **找车/发布** `/api/rides`：搜索、发布、详情、预订  
- **我的行程** `/api/trips`：列表、取消、评价、联系人  
- **个人中心** `/api/profile`：资料、车辆、地点、通知设置  
- **支付 / 通知 / 帮助 / 评价 / 邀请 / 紧急联系人 / 公共 / 上传 / 管理后台** 等  

前端另设 **Mock 模式**（登录页开关），可在无后端时独立演示 UI，降低联调阻塞。

## 三、详细设计方案

### 3.1 后端分层设计（本人重点落实）

后端严格遵循四层结构，**禁止跨层调用**：

```
Router → Controller → Service → DAO
         ↑
    Middleware（auth / admin-auth / optional-auth）
```

| 层级 | 职责 | 示例（本人开发的认证模块） |
|------|------|---------------------------|
| Router | 仅定义路径与方法 | `auth-router.js` 注册 16 个认证相关路由 |
| Controller | 接收参数、调用 Service、封装响应 | `auth-controller.js` 校验 phone/password 格式 |
| Service | 核心业务逻辑 | `auth-service.js` 密码校验、JWT 签发、注册事务 |
| DAO | 数据库读写 | `user-dao.js` 通过 Prisma 操作 `auth_users` |
| Middleware | 鉴权与守卫 | `auth-middleware.js`、`admin-auth-middleware.js` |

**统一响应格式**（全项目强制）：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "requestId": "RN-xxx"
}
```

**接口方法约定**：查询用 GET，新增/修改/删除一律 POST（符合课程与团队规范，避免 PUT/DELETE 混用）。

### 3.2 认证模块详细设计（本人实现）

#### 3.2.1 注册流程

采用「昵称校验 → 阿里云发码 → 验码获 tempToken → 提交注册」四步串行流程，防止跳过短信验证直接注册：

```
App → GET  /api/auth/register/check-nickname
App → POST /api/sms/send
App → POST /api/sms/verify        （返回 tempToken）
App → POST /api/auth/register     （须携带 tempToken）
```

密码规则：8～20 位，须同时包含字母与数字；密码以 bcrypt/sha256 哈希存入 `auth_users.password_hash`。

#### 3.2.2 登录与 Token

- `GET /api/auth/login/config`：按平台返回登录页动态配置（第三方入口等）。  
- `POST /api/auth/login/password`：校验手机号与密码，签发 `token` / `refreshToken`；`rememberMe=true` 时延长有效期。  
- `POST /api/auth/refresh`：刷新访问令牌，减少频繁登录。  
- 禁用账号（`status=disabled`）在登录层直接拦截，返回 401。

#### 3.2.3 找回密码

`check-phone` → `password/sms` → `password/verify-code` → `password/reset` 四接口闭环，与注册发码逻辑分离，重置场景走 Redis 自管验证码。

### 3.3 管理后台详细设计（本人实现）

采用**单表扩展方案**：在 `auth_users` 增加 `role`（user/admin）与 `status`（active/disabled），不另建管理员表，与现有登录链路完全兼容。

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/admin/users` | GET | 分页查询，支持 phone/role/status 筛选 |
| `/api/admin/users/status` | POST | 启用/禁用用户 |
| `/api/admin/users/role` | POST | 提升/降级角色 |

- 所有管理接口经 `admin-auth-middleware` 校验 JWT 且 `role=admin`。  
- 每次改状态/改角色写入 `admin_audit_logs` 审计表，记录操作人、目标用户、IP、时间，满足答辩「操作可追溯」展示需求。

### 3.4 数据库设计概要

全库 **39 张表**，由 Prisma Schema 统一管理。本人直接相关的核心表：

| 表名 | 用途 |
|------|------|
| `auth_users` | 登录账号、角色、状态 |
| `admin_audit_logs` | 管理员操作审计 |
| `real_name_auths` | 实名认证（与 users 模块协作） |

完整 ER 与字段说明见项目 `docs/backend/数据库设计文档.md`（成员 C 主笔，本人参与评审）。

### 3.5 部署架构详细设计（本人实现）

测试环境部署在**腾讯云**，流程如下：

1. 本地 `npm run build` 生成 `dist/`（含 src、package.json，不含 node_modules/.env）。  
2. 上传至服务器 `/var/www/carpooling-api`。  
3. 服务器执行 `npm install --production`、`npx prisma migrate deploy`。  
4. PM2 守护进程：`pm2 start src/index.js --name carpooling-api`。  
5. Nginx 将公网域名反代至 `127.0.0.1:3000`，前端 `EXPO_PUBLIC_API_URL` 指向该域名。  
6. 健康检查：`GET /health` 返回 `db_connected: true` 表示服务与数据库均正常。

### 3.6 前端协作设计要点

虽非本人主责编码，但作为负责人制定了以下前端约束（写入 `CONTRIBUTING.md`）：

- 统一使用 `src/utils/request.ts`，禁止 API 文件自建 axios 或硬编码 URL。  
- Mock 判断统一读 `useEnvStore.isMockMode`，禁止用 `NODE_ENV` 代替。  
- `requestId` 由 Page 层生成，经 props 注入 Hook，保证日志全链路可追溯。

## 四、编码规范制定与程序编写

### 4.1 编码规范制定

本人与成员 E 共同制定并维护 **`CONTRIBUTING.md`**（根目录）及 **`docs/ops/`** 协作文档，核心内容包括：

#### （1）命名规范

- 变量/函数：小驼峰，函数以动词开头（`getUserList`）。  
- 常量：全大写下划线（`MAX_REQUEST_COUNT`）。  
- 文件/目录：短横线分隔（`auth-service.js`、`user-management`）。

#### （2）日志规范（强制）

五级日志：Debug / Info / Warn / Error / Fatal。每条日志必须含 `requestId`、模块、操作类型、脱敏参数；**禁止**明文打印手机号、密码、Token。

#### （3）后端分层与接口规范

- 严格 Router → Controller → Service → DAO，禁止跨层。  
- 响应统一 `{ code, message, data, requestId }`。  
- 路径不超过 3 级，多词用连字符（`/api/emergency-contacts`）。

#### （4）Git 协作规范

- 分支：`main`（生产）、`develop`（开发汇总）、`feature/xxx`、`fix/xxx`、`hotfix/xxx`。  
- Commit：`<type>(<scope>): <subject>`，如 `feat(auth): 完成密码登录与 JWT 签发`。  
- PR 须说明影响范围与测试要点，至少 1 人 Review。

### 4.3 典型编码实例说明

**密码登录核心逻辑**（`auth-service.js`）设计要点：

1. 按手机号查询 `auth_users`，不存在则统一返回「手机号或密码错误」（防枚举）。  
2. 校验 `status !== 'disabled'`。  
3. 比对 `password_hash`（兼容 bcrypt 与历史 sha256）。  
4. 更新 `last_login_at`、设备信息。  
5. 调用 `jwt-utils` 生成 access/refresh token，按 `rememberMe` 设置过期时间。  
6. 全程打 Info/Error 日志，手机号脱敏为 `138****1234`。

## 七、实习经历、感想与收获

### 7.1 实习经历回顾

本次实习以 5 人团队完成一个接近真实交付标准的拼车 App，周期约 10 周。我作为负责人，经历可概括为四个阶段：

**第一阶段（需求与设计，第 1～3 周）**  
组织讨论需求规格说明书，确定「基础拼车 + 管理后台 + Mock 演示」范围；完成系统架构设计、技术选型文档；与成员 C 评审 39 张表的数据库设计；制定接口路径规范，避免后期联调路径不一致。

**第二阶段（核心开发，第 4～6 周）**  
本人优先完成认证模块与管理后台，打通「注册—登录—JWT—鉴权中间件」主链路；接入阿里云短信；成员 D 并行开发行程/支付模块；成员 B 前端 Mock 与页面对接。此阶段最大挑战是**短信验码与 tempToken 时序**，通过序列图固化流程后联调效率明显提升。

**第三阶段（联调与测试，第 7～9 周）**  
部署腾讯云测试环境，统一 `EXPO_PUBLIC_API_URL`；维护接口汇总清单，按阶段推进联调。**收尾测试分工**：本人负责后端 `npm test`、集成用例与报告后端章节；许铷恩负责前端 `npm run test:ci` 与报告前端章节。最终 `TEST_REPORT.md` 记录前端 242 例、后端 88 例全部通过。

**第四阶段（文档与答辩准备，第 10 周）**  
整理 README 与 docs 文档体系，修正过时描述；准备答辩演示脚本；复盘验收标准勾选情况。

### 7.2 遇到的问题与解决

| 问题 | 解决方式 |
|------|----------|
| 前后端接口路径不一致 | 制定 `/api/` 统一前缀与联调清单，PR 必查路径 |
| 注册绕过短信验证 | 强制 tempToken 一次性消费，register 接口校验 |
| 管理员与普通用户两套登录 | 采用 auth_users 扩展 role 字段，单点登录 |
| 成员进度不一致 blocking | 引入 Mock 模式，前端不依赖后端可并行 |
| 生产环境数据库迁移 | Prisma migrate deploy + 文档化 baseline 流程 |

### 7.3 收获与体会

**技术层面**

1. 掌握了 **Express 四层架构**的落地方式，理解 Controller 薄、Service 厚的分工。  
2. 深入实践 **JWT 双 Token、bcrypt 密码存储、RBAC 简易权限**等安全常见方案。  
3. 熟悉 **Prisma 迁移、MySQL 索引设计、Redis 验证码缓存**等工程化细节。  
4. 完成 **云服务器部署全流程**（打包、Nginx、PM2、环境变量），从「能跑 localhost」到「能公网访问」。  
5. 理解 **单元测试 Mock 分层**对路径覆盖的价值，而非仅追求语句覆盖率数字。

**工程与协作层面**

1. **规范先行**：命名、日志、Commit 规范看似繁琐，但在 5 人并行时显著降低沟通成本。  
2. **文档即契约**：接口汇总清单是联调唯一真相，本人维护该文档减少了大量重复问答。  
3. **负责人不仅是写代码**：任务拆分、PR 审核、风险预案、部署运维同样占用大量时间，但这是项目能按时交付的关键。  
4. **测试左移**：auth-service 测试在联调前发现多处边界问题，证明「写测试」不是答辩凑数，而是实打实提效。

**不足与展望**

1. 部分前端 Hook 覆盖率仍偏低，Forgot Password、Trips 等模块测试待补充。  
2. 后端 router/controller 层自动化覆盖不足，后续可补充 Supertest 接口级测试。  
3. 尚未引入 CI 流水线自动跑测，依赖本地 `npm test`，后续可接入 GitHub Actions。  
4. 地图、支付等模块与第三方深度集成仍有优化空间。
